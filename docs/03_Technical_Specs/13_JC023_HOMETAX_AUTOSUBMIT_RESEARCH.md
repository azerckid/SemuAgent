## 0. 목적 · 범위

[Product Baseline — Strategic Direction](../01_Concept_Design/01_PRODUCT_BASELINE.md)의 JC-023(사용자 승인 기반
홈택스 자동제출)에 대한 Implementation Precondition(조사 과제)의 답을 정리한다. 5개 병렬 조사를 통합:
전자신고 파일 규격 / 국세청 공식 API / 법적 경계 / 인증 자동화 기술·보안 / 실무 SW 제출 흐름.

Related: [Backlog JC-023](../04_Logic_Progress/00_BACKLOG.md) · [Filing Support Pre-Code Brief](./09_FILING_SUPPORT_PRE_CODE_BRIEF.md)(현행 신고지원=중간 단계)

## 1. 핵심 결론 (Executive Summary)

1. **파일 생성은 실현 가능** [확실] — 국세청이 세목별 전자신고 표준 파일 규격을 정의(부가세 K/H/J 형식 등),
   부가세·원천세(지급명세서)·법인세 3세목 모두 "파일변환신고" 제도가 존재하고 시판 SW가 구현 중.
2. **공개 매뉴얼에서 확인되는 표준 제출 경로는 "파일변환신고"** [확실] — 규격 파일 생성 → 홈택스 업로드 →
   형식·내용 검증 → 전자파일제출(공동인증서 서명). **일반 개발자용 무인 REST 제출 API는 공개 조사 기준
   확인되지 않았다** [미확인].
3. **확인한 주요 실무 SW 매뉴얼은 동일한 최종 관문을 공유** [확실] — 더존 Smart A·위하고, 세무사랑,
   **KcLep(국세청/세무사회 배포)** 모두
   "규격 파일 → 홈택스 파일변환신고 → 인증서 서명 제출"을 최종 관문으로 공유. 최종 [전자파일제출하기]는
   사람이 인증서로 서명하는 단계.
4. **완전 무인 자동제출은 공개 조사 기준 공식 채널이 확인되지 않았다** [미확인] — 따라서 현실적·지속가능한 방향은
   **"사용자 승인 기반 self-filing 어시스턴트"**: SW가 규격 파일을 만들어주고, **최종 제출은 사용자가
   본인 인증으로 직접 승인·서명**.
5. **자격증명 원문 미저장 자동화는 기술적으로 가능** [확실] — 간편인증 PUSH(개인키가 사용자 단말에만),
   금융인증서 클라우드, 제출 직전 실시간 승인. 제품 원칙(원문 미저장·사용자 최종 승인)과 정합.
6. **핵심 병목은 기술이 아니라 정책·법규** — 국세청이 스크래핑을 능동 차단(2025.5~)하고 오픈API로 전환 중;
   세무사법상 대리 작성·제출 리스크; 2026.6.24 시행 개정법의 마케팅 표현 규제.
7. **게이트**: ① 파일 규격 **적합성 검정**(정식 제출 자격) ② **법무 검토**(세무대리·약관·개인정보).

## 2. 영역별 조사 결과

### 2.1 전자신고 파일 규격 / 파일변환신고

- 국세청이 세목별 표준 파일 포맷을 정의·개정(귀속연도별). 회계프로그램이 규격에 맞춰 **암호화 파일**(비밀번호
  8~15자리) 생성, 홈택스가 복호화(NTS-CRYPTO)해 읽음. [확실]
- 부가세·원천세(지급명세서)·법인세 **3세목 모두 파일변환신고 가능**. [확실]
- **연말 지급명세서** 전산매체 제출요령은 국세청이 PDF/HWP로 공개 배포(홈택스 자료실, 예: 2025 귀속 게시 1175). [확실]
- **근로소득 간이지급명세서**는 **별도** `간이지급명세서(근로소득) 전산매체 제출요령`이며 연말 지급명세서 묶음(1175)과
  **다른 문서**다. JC-030 Slice 0a에서 입수 경로 확정 — [Layout Acquisition](./28_JC030_SIMPLIFIED_WAGE_EFILING_LAYOUT_ACQUISITION.md). [확실]
- 부가세·법인세 **상세** 레이아웃 규격서의 일반 개발사 공식 입수 경로는 여전히 [미확인].
- 규격은 **매년 개정** → 착수 시점 귀속연도 규격 기준으로 검증 필요. [확실]

### 2.2 국세청 공식 API / 연계 채널

- **일반 개발자에게 공개된 신고 제출용 REST API는 확인되지 않음** [미확인]. "조회"는 **사업자등록
  상태/진위확인 오픈API**(data.go.kr, 국세청 제공)가 공식 오픈API로 확인된다. 제출 API 존재 여부는
  국세청 공식 문의로 확정해야 한다.
- 검색에 뜨는 "전자세금계산서/현금영수증 API"는 국세청 공식이 아니라 **민간 중계사(CODEF/쿠콘 등)** 상용
  서비스 → 공식 채널에서 제외. [확실]
- 전자세금계산서 발급 시스템(ASP/자체 ERP)은 국세청 고시(제2023-17호) 표준·인증 요건 대상. [확실]
- **공공 마이데이터**: 국세청이 **국세증명 10종**(납세·소득금액·부가세 과표 증명 등)을 **개인정보 전송요구권**
  기반으로 제공 → 스크래핑 없이 조회 가능. 단 **"제출"은 아님**. [확실]

### 2.3 법적 경계 (세무사법 · 약관 · 마케팅)

- 세무사법 제2조: "납세자의 **위임을 받아** 신고·신청 대리, **세무서류 작성**" 등이 세무사 직무(세무대리).
  무자격 세무대리는 **3년 이하 징역 또는 3천만원 이하 벌금**. [확실]
- **self-filing 보조 = 안전지대 / 대리 작성·제출 = 리스크** [추정]: 사용자가 본인 명의·의사로 신고하고 SW가
  계산·서식·안내를 보조하면 전통적으로 세무대리로 보지 않음. SW가 **불특정 다수의 신고서를 대신 작성·제출**하면
  무자격 세무대리 소지. "자동 신고서 작성"이 "단순 보조"인지 "작성 대리"인지가 회색지대 → **법무 검토 필수.**
- **삼쩜삼**: 무자격 세무대리 고발 → **무혐의 확정**(2023~), 이후 **세무사 제휴 모델로 전환**. 중개수수료
  구조가 알선 시비의 갈림길. [확실]
- ⚠️ **무자격자 세무대리 표시·광고 금지** [확실·법령]: **세무사법 제20조 제3항** — "제1항에 따라 세무대리를 할
  수 없는 자는 세무대리 업무를 취급한다는 뜻을 표시하거나 광고를 하여서는 아니 된다." 위반 시 **제22조의2
  제10호**에 따라 **1년 이하 징역 또는 1천만원 이하 벌금**. 즉 실제 대리를 안 해도 "세무대리 취급/오인" 표시·
  광고만으로 처벌 대상. → **SemuAgent 마케팅 카피 관리가 직접 리스크.**
  (2026-06-24 시행 및 "세금환급·절세전문·기장대행" 등 구체 오인광고 유형은 시행령·국세청장 고시·보도 기준
  [추정] — 법령 원문은 "세무대리 취급 표시·광고 금지"라는 포괄 규정.)
- 홈택스 "이용에 관한 규정"의 자동화·대리접속 금지 **조문 원문은 미확인** → 착수 전 직접 확인 필요. [미확인]

### 2.4 인증 기반 자동화 기술·보안

- 인증 수단: 공동인증서, **금융인증서(클라우드 보관)**, **간편인증(카카오·네이버·PASS·토스; PUSH 방식 = 개인키가
  사용자 단말에만)**, 디지털원패스. 사업자도 간편인증만으로 전자세금계산서 발급 가능해짐. [확실]
- **국세청 ANTI-스크래핑 능동 차단(2025.5~)** + 오픈API 구축 발주(세무플랫폼 **사전등록 + 이용수수료** 구조).
  → **범용 스크래핑 방식은 지속 불가.** [추정·보도] (국세청 공식 발표문이 아니라 보도 기준 — 정책 방향은 신뢰도
  높으나 세부·시점은 국세청 공식 확인 필요.)
- **자격증명 원문 미저장 설계는 가능하며 국내 표준 패턴** [확실]: 간편인증 PUSH(서버는 결과 토큰만) · 금융인증서
  클라우드 · 제출 직전 실시간 사용자 승인. **인증서 개인키·비번을 서버 보관하는 방식은 접근매체 위임 금지·고유
  식별정보 규제에 직결 → 채택 불가.** [확실]

### 2.5 실무 SW 제출 흐름 (더존 · 위하고 · KcLep)

- **확인한 주요 실무 SW 매뉴얼 기준 동일한 정식 제출 관문** [확실]: 프로그램에서 암호화 전자신고 파일 생성 → 홈택스 **[신고/납부]→세목→
  [파일변환신고]** 업로드 → **형식검증 → 내용검증 → 전자파일제출(공동인증서 서명)** → 접수증('정상') 확인.
- **인증이 2단계로 분리** [확실] — ① **자료수집(스크래핑)용**: 홈택스 인증서를 프로그램에 등록해 계산서·현황
  조회. ② **제출용**: 홈택스 로그인 + 최종 [전자파일제출하기] 서명. *SemuAgent 설계의 중요한 구분점.*
- **세무대리인 경로**: 납세자가 홈택스에서 **수임동의** → 세무사가 자기 인증서로 로그인해 **파일변환신고로 대량
  제출**. [확실]
- **KcLep = 국세청 인정 표준 흐름의 기준점**. 국세청은 **적합성 검정을 받은 세무프로그램(약 15개 업체)**이 만든
  파일만 파일변환신고로 받음 → **적합성 검정 통과 = 정식 제출 자격의 관문.** [확실/일부 추정]

## 3. 실현가능성 판정 (단계별)

| 단계 | 공식·합법 경로 | 실현가능성 |
|---|---|---|
| 신고자료 조회 | 사업자상태 오픈API · **공공 마이데이터(국세증명 10종)** | 가능(제한적) [확실] |
| 신고서 계산·작성 | SW 내부 계산 (제품 원칙: 사용자 검토 강제) | 가능 [확실] |
| 전자신고 **파일 생성** | 국세청 표준 규격 + **적합성 검정** | 가능(검정 전제) [확실] |
| **파일 제출** | 홈택스 **파일변환신고** 업로드 + 인증서 서명 | 가능하나 **사용자/세무사 인증 서명 필수** [확실] |
| **완전 무인 자동제출** | 일반 공개 제출 API는 확인되지 않음 | 공식 문의 전 구현 판단 보류 [미확인] |
| 접수증 회수·보관 | 제출 후 홈택스 접수증 | 가능 [추정] |

## 4. 권고 아키텍처 방향 (제품 원칙 정합)

1. **조회는 스크래핑 대신 공식 경로**: 공공 마이데이터(전송요구권) + 국세청 오픈API(개시 시). 정책 리스크 제거.
2. **파일변환신고를 최종 관문으로 전제**: SemuAgent가 **적합성 검정 규격** 전자신고 파일을 생성. (확인한 주요
   실무 SW의 길 = 현재 가장 안전한 벤치마크.)
3. **제출은 사용자 승인 기반 self-filing**: 사용자가 본인 인증으로 파일변환신고 업로드·서명. SW는 파일 생성·안내·
   사전입력까지, **최종 제출 서명은 사용자**. "완전 무인"이 아니라 **"원클릭 준비 + 사용자 서명 제출"**.
4. **자격증명 원문 미저장**: 간편인증 PUSH / 금융인증서 클라우드 / 제출 직전 실시간 승인. 세션 최소 수명·즉시 폐기.
5. **감사 로그·동의 이력**: 전송요구·제출 승인 이력 보관(개인정보 안전성 확보조치 기준).
6. **마케팅 표현 관리**: "대행·대리·환급·절세전문·기장대행" 금지. **"직접 신고를 돕는 self-filing 도구"**로 포지셔닝.

## 5. JC-023 비전 재구성 (지금 가능 vs 공식 API 대기)

- **지금 실현 가능(로드맵 1단계)**: 원클릭 신고 **준비**(자료 정규화 → 계산 → 검증 → 규격 파일 생성) +
  **사용자 승인 기반 제출 가이드/사전입력**. 사용자가 본인 인증으로 마지막 서명.
- **공식 API 의존(로드맵 2단계, 미확정)**: 국세청 오픈API의 **제출 지원 범위**가 열리고 세무플랫폼 등록·수수료
  조건이 확인되면, 더 높은 자동화 수준 검토. 현시점 [미확인].
- 즉 "**완전 무인 원클릭 자동제출**"은 최종 지향점이되, **현재 공개 조사 기준으로는 사용자 서명 관문이 남는**
  self-filing 자동화가 실현 가능한 최대치다. 제출 API·연동 채널은 국세청 공식 문의 전까지 [미확인]으로 둔다.
