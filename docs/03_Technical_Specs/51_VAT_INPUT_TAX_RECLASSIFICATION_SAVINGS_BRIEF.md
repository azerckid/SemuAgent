## 0. Decision

기장 단계에서 매입 거래가 법정 불공제 사유(현재 범위: 접대비/기업업무추진비)로
분류되면, 그 이후 부가세 판단은 그 분류를 그대로 따라간다. 그러나 분류 자체가
기장 실수였을 수 있다 — 같은 "식대" 지출이라도 참석자가 전부 내부 직원이면
복리후생비·회의비이고, 그 경우 매입세액을 공제받을 수 있다. 지금은 이 가능성을
아무도 다시 확인하지 않는다.

JC-041은 **이미 불공제로 분류된 매입 중, 재분류하면 사업자가 실제로 세금을
아낄 수 있는 거래를 찾아 제안**한다. 목표는 "판단이 맞는지 확인해달라"가 아니라
"**놓치고 있는 절세 기회를 최대한 찾아서 보여준다**"는 것이다.

### 0.1 핵심 원칙: 후보는 넓게, 확정은 엄격하게

v1(최초 작성)은 근거가 부족하면 후보 자체를 표에서 제외하는 이진 게이트
모델이었다. 리뷰 결과 이 모델은 "찾아서 보여준다"는 목적과 맞지 않는다는
지적을 받았다 — 참석자 정보가 없다는 이유만으로 후보 자체가 안 보이면,
사용자는 정보를 보완해서 공제를 받을 수 있는 기회가 있었다는 사실 자체를
영영 알 수 없다.

v2는 **"어떤 것을 사용자에게 보여줄지"와 "어떤 것을 실제로 공제로 바꿀 수
있는지"를 분리**한다.

- **후보 목록(넓게)**: 접대비로 분류된 불공제 후보는 근거의 강약과 무관하게
  전부 목록에 올린다. 근거가 약하거나 없어도 후보에서 제외하지 않는다 —
  대신 신뢰도를 "낮음"으로 표시하고, 공제로 전환하려면 무엇이 더 필요한지
  함께 보여준다.
- **확정(엄격하게)**: 실제로 접대비 → 복리후생비/회의비로 재분류해 매입세액
  공제를 적용하는 것은 여전히 엄격하다. AI가 자동으로 전환하지 않으며,
  사용자가 부족한 근거를 보완하고 명시적으로 확정해야만 canonical 값이
  바뀐다.

즉 §4.2의 "부정 근거"들(참석자 정보 없음, 고액, 법인 거래처명 등)은 더 이상
**후보 제외 조건**이 아니라 **신뢰도를 낮추는 감점 요인**이다. 근거 없이
후보를 만들지 않는다는 원칙은 유지하지만, "근거 없음"의 의미가 "숨긴다"에서
"낮은 신뢰도로 보여주고 부족한 자료를 요청한다"로 바뀐다. 부당공제 리스크는
확정 단계의 엄격한 게이트로 막는다(§5).

## 1. Origin and Motivation

2026-07-12 VUI-1b(부가세 화면 단순화) Preview 리뷰 세션 중 프로젝트 오너가
"확인 필요 거래" 표에 남아있는 불공제 판단(접대비)이 사실은 이미 확정된
사실이라 재확인이 불필요하다는 점을 지적했다. 이어서 "공제 못 받는 것으로
되어 있는 것 중 실제로는 공제받을 수 있는 걸 찾아주면 사업자에게 이득이
되지 않겠냐"는 방향 제안이 나왔고, 이 문서로 정리했다.

v1 구현(PR #221) 리뷰 과정에서 "목적은 최대한 많은 후보를 찾아서 보여주는
것"이라는 전제가 다시 확인됐고, 근거가 약하면 후보를 아예 숨기는 v1의 이진
게이트가 그 전제와 맞지 않는다는 결론에 따라 §0.1의 신뢰도 등급 모델로
재설계했다.

핵심 통찰: 부가세법상 공제/불공제는 법이 이미 정한 폐쇄형 분류 체계(제38·39조)라
"AI가 몰라서 물어보는" 영역이 아니다. 반면 **그 거래가 애초에 올바른 카테고리에
분류됐는지는 별개 문제**이며, 여기에는 기장 단계의 실수가 개입할 여지가 있다.
이 실수를 찾아 고치면 사업자에게 직접적인 금전적 이득이 된다.

## 2. Scope

### 2.1 In Scope (1차)

- 부가세 매입세액 불공제 사유 중 **④ 기업업무추진비(접대비) 및 이와 유사한
  비용 관련**으로 분류된 거래만 다룬다. 이 필터는 신뢰도와 무관한 **범위
  경계**이며 §0.1의 "넓게" 원칙과 별개다 — 접대비 외 다른 불공제 사유는
  아직 이 기능의 대상이 아니다(§2.2).
- 재분류 후보: 복리후생비, 회의비 — 둘 다 참석자가 내부 직원으로 한정될 때
  성립하는 카테고리이며, 부가세 매입세액 공제가 가능하다.
- 위 범위 안의 불공제 후보는 **전부** 재분류 후보 목록에 오른다(근거
  강약과 무관, §0.1). 각 후보는 신뢰도 등급(§4)과 부족한 자료 목록을 함께
  가진다.
- 실제 재분류 확정에는 엄격한 조건이 적용된다(§5).

### 2.2 Out of Scope (1차)

- 접대비 외 다른 법정 불공제 사유(③ 비영업용 소형승용차, ⑤ 면세사업 관련
  등)의 재분류 탐지는 후속 논의. 사유별로 근거 자료 성격이 크게 달라 이번
  범위에 포함하지 않는다.
- 부가세 외 세목(소득세·법인세 등)의 절세 가능성 탐지는 범위 밖.
- 재분류 자동 확정. 기장 계정과목을 AI가 직접 수정하지 않는다.
- 원래 불공제 분류 자체가 틀렸는지(즉 애초에 접대비가 아닌데 접대비로
  기장됐는지)를 포괄적으로 재검증하는 감사 기능. 이건 JC-039의 일반 판단
  범위이며, JC-041은 "불공제 판정을 받아들이되 재분류 여지만 본다."

## 3. Why This Is Different From "확인 필요" (JC-039 대비)

JC-039의 "확인 필요"는 **분류 자체가 불확실**할 때 쓴다(예: 이 매출이
영세율인지 판단할 근거가 부족함). JC-041은 분류가 이미 확정된 상태에서
**"혹시 이 확정이 원래 잘못된 게 아니었나"**를 다시 보는 것이다. 이 둘은
화면에서 섞이면 안 된다 — 불공제 chip을 클릭했을 때 "이미 확정된 사실
설명"과 "혹시 재분류하면 이득" 제안이 뒤섞이면, 사용자가 이미 끝난 판단을
다시 의심해야 하는 것처럼 보인다. UI Contract(§8)에서 이를 분리한다.

## 4. Reclassification Confidence Model

### 4.1 요인(Factor)

각 후보는 다음 요인들을 근거로 신뢰도를 매긴다. §4.2의 반대와 달리, 이
요인들은 후보를 만들거나 없애는 게이트가 아니라 **점수 가산/감산 요인**이다.

**가산 요인(재분류를 뒷받침)**

| 요인 | 설명 | 가중치 |
|:---|:---|:---|
| 참석자 전원 내부 직원 | 적요에 명시된 참석자가 `employee_profile` 활성 직원 명단과 전부 일치 | 강 |
| 내부 행사 적요 키워드 | "회식", "워크샵", "전 직원", "팀 미팅" 등 | 중 |
| 과거 유사 재분류 이력 | 같은 거래처의 과거 기간 결정이 이 거래처는 실제로 공제 전환된 적이 있음을 시사 | 약(참고용, §4.3) |

**감산 요인(재분류에 불리하지만 후보를 제외하지 않음)**

| 요인 | 설명 | 가중치 |
|:---|:---|:---|
| 참석자 정보 없음 | 적요에서 참석자를 특정할 수 없음 | 무점수(정보 부재는 불리한 증거가 아니다) + "부족한 자료"에 추가 |
| 고액 | 반복적인 소액 식대 패턴보다 금액이 큼(운영 임계값, 법정 기준 아님) | 감점 |
| 외부 거래처 특정 표현 | 적요·거래처명에 "거래처", "고객사", "바이어" 등 명확한 외부 신호(모호한 내부 직함은 신호로 안 씀, §4.4) | 감점 |
| 과거 접대비 유지 이력 | 같은 거래처를 과거에 사용자가 명시적으로 접대비로 재확정한 적이 있음을 시사 | 감점(참고용, §4.3) |

### 4.2 신뢰도 등급 산출

가산/감산 요인을 합산해 세 등급 중 하나로 분류한다. 정확한 점수 산식은
구현 시(VAI-9a) 정의하되, 방향은 다음을 따른다.

- **높음**: 강한 가산 요인(참석자 전원 내부 확인)이 있고 뚜렷한 감산 요인이
  없음.
- **중간**: 중간 가산 요인(내부 행사 키워드) 또는 약한 참고 신호만 있음.
- **낮음**: 가산 요인이 없거나 감산 요인이 우세함 — 그래도 접대비 불공제
  후보라는 이유만으로 목록에서 제외하지 않는다.

등급과 무관하게, 모든 후보에는 "왜 후보인지"(발견된 요인)와 "무엇을
추가하면 신뢰도가 올라가는지"(부족한 자료)가 함께 표시된다.

### 4.3 과거 결정 이력은 약한 참고 신호

같은 거래처의 과거 `vat_deduction_review` 결정(재분류로 확정됐는지, 접대비로
유지됐는지)은 **강한 증거로 쓰지 않는다** — 거래 성격은 시점마다 다를 수
있고(예: 같은 식당에서 어떤 날은 팀 회식, 어떤 날은 거래처 접대), 과거 한
번의 판단이 이번 거래를 결정짓지 않는다. 상충하는 과거 이력(재분류 이력과
유지 이력이 둘 다 있음)이 있으면 보수적인 쪽(접대비 유지)을 참고 신호로
쓴다 — 이는 등급을 낮추는 데는 쓰되, 후보 자체를 없애지 않는다.

### 4.4 외부 신호의 모호성 처리

"대표", "이사", "팀장" 같은 표현은 내부 직원의 직함이기도 하므로(예: 대표
1인 회사의 "김대표") 외부 거래처 신호로 쓰지 않는다. 참석자 명단 대조가
더 신뢰할 수 있는 내부/외부 판별 수단이므로, 모호한 직함 키워드에 의존하지
않는다. "거래처", "고객사", "바이어"처럼 내부 직함과 겹치지 않는 표현만
감산 요인으로 쓴다.

### 4.5 staffMemo 포함

`bookkeeping_transaction_classification.staffMemo`(담당자가 직접 남긴 메모)는
`vat_deduction_review.description`과 함께 근거 탐색 대상 텍스트에 포함한다.
담당자가 참석자·목적을 기장 시 메모로 남겼다면 이것이 가장 신뢰할 수 있는
근거이므로 놓치면 안 된다.

## 5. Confirmation Gate (엄격)

후보 노출과 실제 확정은 분리된다. 후보는 신뢰도와 무관하게 전부 보이지만,
**공제로 실제 전환하려면(canonical 값 변경) 다음을 모두 만족해야 한다.**

1. 참석자(또는 업무 목적)를 사용자가 명시적으로 확인·입력했다.
2. 적격증빙(세금계산서·카드 내역 등 기존에 이미 연결된 증빙)이 있다.
3. 사용자가 재분류를 명시적으로 확정했다(AI가 자동으로 전환하지 않음).

신뢰도가 "낮음"인 후보는 위 조건을 만족하지 못하는 상태이므로, 사용자가
부족한 자료(§4의 "부족한 자료" 목록)를 보완하기 전까지 확정 버튼이 비활성
상태이거나, 확정 시 그 자료를 먼저 입력하도록 요구한다. 정확한 UI 동작은
VAI-9d/9e에서 확정한다.

"접대비 유지"를 선택하면 같은 거래·같은 패턴에 대해 반복해서 다시 묻지
않는다(사용자 결정을 감사 이력으로 저장, §4.3 참고 신호로 재사용 — 단
강제 게이트로는 쓰지 않는다).

## 6. Tax Savings Framing (제안 문구 요건)

제안에는 반드시 다음이 함께 있어야 한다.

- 현재 분류와 제안 분류(예: "접대비 → 복리후생비 후보")
- 신뢰도 등급(높음/중간/낮음)과 그 근거(어떤 요인이 발견됐는지)
- **금액은 확정형이 아니라 가능성형으로 표시**: "132,000원 절세 확정"이
  아니라 "**최대 132,000원 추가 공제 가능성**" — §5의 확정 조건을 충족하는지
  않은 상태에서 금액을 기정사실처럼 보여주면 안 된다.
- 확정에 필요한 조건과, 지금 부족한 자료가 무엇인지(§4의 "부족한 자료")
- 재분류가 최종이 아니며 사용자 확인이 필요하다는 명시

신뢰도 "높음"이라도 §5의 확정 게이트를 통과하기 전까지는 "가능성" 표현을
유지한다.

## 7. Data Contract

기존 `vat_deduction_review`(사용자 확정 감사 테이블)는 최종 분류 결정을
저장하는 곳이다. VAI-9c의 재분류 제안은 원천 행에서 매번 결정론적으로 다시
만드는 read model이므로 신규 DB에 중복 저장하지 않는다. 사용자 결정과 감사
참조는 VAI-9e에서 기존 정본과 원자적으로 저장한다.

| 필드 | 목적 |
|:---|:---|
| evaluation.confidence | 신뢰도 등급(`high`/`medium`/`low`) |
| currentCategory / evaluation.suggestedCategory | 현재 분류(접대비) / 제안 분류(복리후생비·회의비, 신호가 전혀 없으면 null) |
| evaluation.factors | §4.1 가산/감산 요인 목록(각 요인의 방향과 요약) |
| evaluation.missingToConfirm | 확정하려면 필요한 자료 목록(§5) |
| potentialSavingsKrw | 재분류 시 추가로 공제 **가능한**(확정 아님) 매입세액 |
| savingsBasis | `maximum_additional_input_tax_if_fully_reclassified`; 총액 역산이 아니라 저장된 매입세액을 사용했다는 계산 근거 |
| userDecision | `pending` / `reclassified` / `kept_as_is` |
| decisionRowId | 사용자 결정을 저장하는 감사 row 참조(재질문 방지용) |

정본은 여전히 `vat_deduction_review`의 사용자 확정값이며, 이 필드들은
추천·이력 계층으로 canonical 값을 대체하지 않는다.

VAI-9c 계산 규칙은 하나뿐이다.

- `potentialSavingsKrw = inputTaxKrw`
- 공급가액·합계액에서 세액을 역산하지 않는다.
- `inputTaxKrw = 0`이면 가능 금액도 0원이며 이익을 만들어내지 않는다.
- Zod 계약은 두 금액이 다르면 결과를 거부한다.

## 8. UI Contract (VAI-9d Preview 승인·VAI-9e runtime 반영)

- 재분류 제안은 세액 요약 바로 아래의 `추가 공제 가능성` 섹션에 두고,
  `신고 전 수정 필요` 작업대와 **분리**한다. 불공제 설명 팝오버에 절세 제안을
  섞지 않는다.
- 기본 행에는 거래/상대처, `접대비 -> 제안 분류`, `최대 OOO원 추가 공제
  가능성`만 표시한다. 신뢰도는 작은 보조 텍스트로 두고 badge를 늘리지 않는다.
- 행을 펼치면 `왜 후보인가`, `확정 전 필요`, 가능 금액이 확정 절세액이
  아니라는 주의 문구를 표시한다.
- 높은/중간/낮은 신뢰도 후보를 모두 같은 목록에 표시한다. 후보가 0건이면
  섹션 자체를 렌더하지 않는다.
- VAI-9d Preview 승인 후 VAI-9e runtime은 펼쳐보기 안에만 `공제로 재분류`와
  `접대비 유지`를 둔다. 기본 행에는 저장 control을 추가하지 않는다.
- `공제로 재분류` 모달은 복리후생비/회의비 선택, 업무 목적 또는 참석자 입력,
  연결된 적격증빙 상태를 한 번에 확인한다. 적격증빙이 없으면 확정 버튼을
  비활성화하며 서버도 같은 조건을 다시 검증한다.
- `접대비 유지`는 선택적 사유와 함께 명시적으로 확정하며, 저장 후 해당 거래는
  후보 목록에서 제외해 반복 질문하지 않는다.
- 후보 조회는 세액 요약과 예외 작업대의 초기 렌더를 막지 않는 별도 Suspense
  경계에서 실행하고, 과거 거래처 결정은 후보별 N+1이 아닌 단일 묶음 조회로 읽는다.
- 제안 화면은 [03_vat.html](../02_UI_Screens/previews/03_vat.html)에 반영했으며
  실제 runtime 구현 전 프로젝트 오너의 화면 승인이 필요하다.

## 10. Acceptance Criteria

- [x] 접대비 불공제 후보는 신뢰도와 무관하게 전부 목록에 오른다(§0.1) — 근거
      부족을 이유로 후보에서 제외하지 않는다.
- [x] 모든 후보에 신뢰도 등급, 발견된 요인, 부족한 자료 목록이 함께 표시된다.
- [x] 금액은 확정형이 아니라 "최대 OOO원 가능성" 형태로만 표시된다.
- [x] 실제 공제 전환(canonical 값 변경)은 §5의 확정 게이트를 통과해야만
      가능하다 — 신뢰도가 낮은 후보는 자료 보완 없이 확정할 수 없다.
- [x] AI가 계정과목이나 공제 여부를 자동으로 확정하지 않는다.
- [x] 사용자가 "접대비 유지"를 선택하면 같은 거래·패턴에 재질문하지 않되,
      이 결정을 강제 게이트가 아니라 참고 신호로만 재사용한다(§4.3).
- [x] UI에서 "이미 확정된 불공제 설명"과 "재분류 후보"가 서로 다른 자리에
      노출된다.
- [x] 접대비 불공제 후보가 0건이면 관련 UI 요소가 아예 나타나지 않는다.
- [x] tenant·사업장·기간 격리와 PII 최소화를 유지한다.
- [x] 대표 fixture(높음/중간/낮음 각각, 그리고 상충하는 과거 이력 케이스)로
      등급 산출과 확정 게이트 분리를 검증한다.
