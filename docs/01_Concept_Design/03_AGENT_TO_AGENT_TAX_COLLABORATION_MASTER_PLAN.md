# Agent-to-Agent Tax Collaboration Master Plan

## 1. Purpose

이 문서는 사업자 측 **SemuAgent**와 세무회계사무소 측 **JARYO-GIWA**를 연결하는
Agent-to-Agent 세무 협업의 상위 제품 계획을 정의한다.

이 문서는 구현 명세가 아니다. 현재까지 사용자와 합의한 제품 원칙을 기록하고,
남은 결정을 `A2A-0`부터 `A2A-7`까지 질의응답 단위로 나누어 확정하기 위한
Master Plan이다.

## 2. North Star

```text
사업자: SemuAgent로 자료 수집·대조·세무 판단·신고 준비
  -> 기본 경로: 사업자가 홈택스·위택스에서 직접 신고
  -> 선택 경로: 준비 완료 자료를 계약된 파트너 사무소에 전달
세무회계사무소: JARYO-GIWA로 정제된 자료를 수신·검토
  -> 세무대리인이 최종 판단
  -> 검정 SW로 신고
```

SemuAgent는 사업자가 직접 신고할 수 있도록 자료를 정리하고 안내하는 것이 기본이다.
직접 신고가 번거로운 사업자에게는 선택적으로 `세무회계사무소 연결`을 제공한다.

JARYO-GIWA를 사용하는 사무소는 이미 수집·대조·분류·확정된 자료를 받는다. 따라서
자료 요청, 누락 확인, 형식 통일, 재입력, 반복 문의에 쓰는 시간을 줄이고 세무 판단과
최종 신고에 집중할 수 있다.

## 3. Product Value

### 3.1 사업자

- 직접 신고를 기본 선택으로 유지한다.
- 원할 때만 계약된 세무회계사무소에 준비 완료 자료를 전달한다.
- 사무소에 맡기더라도 같은 자료를 다시 모으거나 설명하지 않는다.
- 전달 범위와 수신자를 확인한 뒤 직접 전송한다.

### 3.2 세무회계사무소

- 원본자료와 SemuAgent가 정리한 신고 준비 자료를 함께 받아 대조한다.
- 자료 수집·정리·재입력 업무를 크게 줄인다.
- 사업자 사실 확인이 필요한 예외만 구조화해 되돌려보낸다.
- 더 적은 반복 작업으로 더 많은 고객을 안정적으로 처리할 수 있다.

### 3.3 두 제품

- SemuAgent는 사업자의 세무 준비 에이전트다.
- JARYO-GIWA는 세무회계사무소의 업무 수행 에이전트다.
- 두 에이전트는 사람의 승인 아래 검증된 자료, 상태, 보완 요청과 처리 결과를 주고받는다.
- 사업자에게는 신고 선택권을, 사무소에는 신규 수임 기회와 낮은 처리비용을 제공한다.

## 4. Approved Product Principles

현재 질의응답으로 다음 원칙을 **제품 방향 수준에서** 승인했다. 상태·필드·문구·예외 처리의
세부 계약까지 승인된 것은 아니며, 단위별 상세 승인 상태는 §10을 따른다.

1. **직접 신고가 기본이다.** 세무회계사무소 연결은 선택 기능이다.
2. **초기 파일럿은 가상 파트너 사무소 1곳**으로 설계한다.
3. 실제 연결 출시 전에는 JARYO-GIWA 사용 사무소와 별도 사업 계약이 선행되어야 한다.
4. 신고 일정과 대상 기간은 법정 일정에 따라 SemuAgent가 결정한다.
5. SemuAgent는 해당 세목·기간 자료를 자동으로 준비하지만 **자동 전송하지 않는다.**
6. 사업자가 수신자, 세목, 기간, 포함 자료와 동의 범위를 확인하고 전송 버튼을 눌러야 한다.
7. 처리 방향이 정해지지 않은 미확정 항목과 치명적 무결성 오류가 없는 **Ready 패키지만**
   사무소에 전송한다.
8. JARYO-GIWA는 Ready 패키지를 최종 검토하며 자동으로 신고 확정하지 않는다.
9. 세무대리인이 전문 범위에서 결정 가능한 사항은 사무소가 처리하고 근거를 남긴다.
10. 사업자만 확인할 수 있는 사실이 필요하면 SemuAgent로 구조화된 수정 요청을 반환한다.
11. 홈택스 제출은 사무소와 검정 SW가 수행한다. 두 에이전트 모두 자동 신고하지 않는다.
12. 연결된 사무소 전송은 **SemuAgent에서 JARYO-GIWA로 직접 연동**하는 것을 기본으로 한다.
13. JARYO-GIWA에 수신된 자료는 해당 사업자 화면에 자동 등록하되 최초 상태는 **검토 대기**다.
14. ZIP 다운로드·수동 업로드는 비연동·장애 상황의 보조 경로이며 A2A 주 경로가 아니다.

## 5. Ready Handoff Principle

Agent-to-Agent 연결의 핵심은 `보냈다`가 아니라 `다시 정리할 필요가 없는 상태로 보냈다`다.

Ready는 사업장 전체가 아니라 **세목·신고기간별**로 판정한다. 한 세목의 미확정 항목이
다른 세목의 Ready 패키지 전송을 막지 않는다.

Ready는 최소한 다음 조건을 만족해야 한다.

- 대상 신고 세목과 기간이 확정됨
- 필수 원천자료 수집 또는 자료 부재에 대한 처리 방향 확정
- 누락·중복·금액 불일치 해결 또는 허용된 예외 처리 완료
- 증빙 연결 완료
- 계정항목과 거래 목적 확정
- 세목별 공제·과세유형 등 사용자 확인 완료
- stale 계산, 합계 불일치, 손상된 데이터, 필수 주체 정보 누락 등 치명적 무결성 오류 0건
- 사업자가 전달 내용을 최종 확인함

자료가 없다는 사실만으로 Ready를 영구 차단하지 않는다. 사용자가 `증빙 없음`, `공제 제외`,
`이번 신고에서 제외`처럼 허용된 처리 방향을 명시적으로 확정하면 해당 항목은 해결된 예외로
간주한다. 이 결정은 이유·확인자·확인시각과 함께 패키지에 포함한다.

반대로 매출 누락 가능성, 산술 불일치, stale 계산, 데이터 손상처럼 신고 결과를 신뢰할 수 없게
만드는 문제는 사용자의 위험 수용만으로 해제할 수 없다. 전송 화면과 서버가 같은 Ready 계약을
재검증한다.

세목별 정확한 Ready 조건은 `A2A-2` 질의응답에서 별도 완료 계약으로 세분화한다.

### 5.1 Transmission Unit

A2A의 정본 전송 단위는 **사업자 1곳 × 신고 세목 1개 × 신고기간 1개 × snapshot version 1개**다.
같은 마감일의 여러 세목을 화면에서 함께 선택할 수는 있지만, 내부적으로는 세목별 독립 전송
이벤트를 생성한다.

각 전송 이벤트는 독립된 다음 정보를 가진다.

- transfer ID와 contract version
- 신고 세목·신고기간
- Ready snapshot·source fingerprint
- 사용자 전송 승인
- JARYO 수신·검토·보완·기술 오류 상태
- 재전송 version과 감사 이력

한 세목의 blocker·보완 요청·기술 오류가 같은 날 선택한 다른 세목의 수신 승인에 영향을 주지
않는다. 여러 세목을 하나의 acceptance 결과로 묶지 않는다.

### 5.2 Source Scope

각 전송에는 해당 세목·신고기간의 Ready 판단과 신고 준비값 생성에 **실제로 사용된 자료만**
포함한다.

포함 범위:

- 신고 준비 행이 참조하는 세금계산서·카드·현금영수증·통장·급여 등 원본자료
- 해당 신고 이벤트에 포함된 정규화 행
- 증빙 연결, 계정 분류, 공제·과세유형과 사용자 확정
- 해결된 예외와 사유·확인자·확인시각
- Ready snapshot, validation 결과, provenance·fingerprint·hash

제외 범위:

- 다른 신고 세목·다른 신고기간의 자료
- 회사의 전체 과거 거래·전체 문서 보관함
- 해당 snapshot이 참조하지 않는 초안·폐기·대체된 자료
- 홈택스·위택스·은행·카드 자격증명과 비밀정보

하나의 원본이 여러 신고 이벤트의 근거로 실제 사용되면 각 이벤트가 그 source reference를
가질 수 있다. 물리 파일은 source ID·hash로 중복 전송을 줄일 수 있지만, 권한과 감사 범위는
각 전송 이벤트별로 독립 유지한다.

### 5.3 Pre-Send User Confirmation

사업자는 전송 직전에 별도 확인 화면에서 최소한 다음 항목을 확인한다.

- 수신 세무회계사무소와 현재 연결 상태
- 대상 사업자, 신고 세목, 신고기간, snapshot version
- 신고 준비값 요약
- 자료유형별 포함 원본 건수와 펼쳐볼 수 있는 상세 원본 목록
- 해결된 예외와 각 처리 방향
- Ready 상태와 미해결 blocker 0건

최종 CTA는 `세무사무소로 전송`으로 표시한다. 사용자가 확인한 fingerprint와 서버의 현재
fingerprint가 다르거나 Ready가 해제되면 전송을 차단하고 최신 내용 재확인을 요구한다.
전송 감사 이력에는 사용자, 확인시각, 수신 사무소, 세목·기간, snapshot version, fingerprint,
포함 source scope를 남긴다.

이 확인은 현재 자료를 실제로 전송하는 **행위 승인**이다. 개인정보 처리 역할과 법적 자료전달
동의 문구는 A2A-5에서 별도로 확정하며, 그 결과가 이 화면의 동의 문구 선행조건이다.

### 5.4 Ready Acceptance Gap

SemuAgent가 Ready로 판정한 패키지를 JARYO-GIWA가 자료 누락이나 무결성 문제로 반송하면,
이를 정상 운영 예외로 취급하지 않는다. **생산자 Ready 기준과 수신자 acceptance 기준 사이의
계약 간극**으로 기록하고 원인을 분류해야 한다.

A2A-2와 A2A-3에서는 최소한 다음 반송 사유를 구분한다.

- SemuAgent Ready false positive
- schema·contract version 또는 hash 무결성 실패
- Ready 이후 원천자료 변경·stale 발생
- 사무소 검토에서 새로 드러난 사업자 사실 확인 필요

포괄적인 `자료 누락` 한 가지로 기록하지 않는다. Ready 패키지 반송은 Ready 완료 계약 개선과
첫 수신 승인율 지표에 반드시 되먹임한다.

## 6. Filing-Schedule-Driven Packages

패키지는 임의 월간 묶음이 아니라 법정 신고 일정의 세목·기간 단위로 준비한다. 화면의 일괄
선택은 편의 기능일 뿐이며, 전송·Ready·감사·반송 정본은 항상 독립 이벤트로 유지한다.

| 신고 이벤트 | 준비 대상 예시 | 정확한 계약 상태 |
|---|---|---|
| 원천세 | 해당 귀속·지급월 급여와 원천세 준비값 | A2A-2에서 확정 |
| 부가세 | 해당 과세기간 매출·매입·공제·과세유형 | **첫 구현·검증 세목 Owner Approved**, 세부 Ready 계약 대기 |
| 지급명세서 | 반기·연간 지급자료 | A2A-2에서 확정 |
| 연간 신고 | 종합소득세·법인세 대상 연도 자료 | 해당 기능 완료 후 확정 |

SemuAgent는 신고 일정에 맞춰 패키지를 미리 준비하고, Ready가 되기 전에는 전송 버튼을
활성화하지 않는다.

### 6.1 First Implementation Tax Type

A2A 직접 연동의 첫 구현·브라우저 E2E·파트너 fixture 검증 세목은 **부가세**다. 부가세는
세금계산서·카드·현금영수증·통장 원본, 자료대조, 공제·불공제·안분과 과세유형 사용자 결정,
Ready gate를 함께 사용하므로 두 제품의 연결 가치를 가장 넓게 검증할 수 있다.

부가세 E2E 완료 후 같은 versioned 전송·수신 계약을 원천세, 지급명세서와 다른 신고 이벤트로
확대한다. 이 결정은 최종 지원 범위를 부가세로 제한하지 않는다.

### 6.2 VAT Ready Contract

첫 부가세 A2A slice는 별도의 Ready 공식을 새로 만들지 않고 현재 VAT package gate의
`isReady`를 단일 원천으로 재사용한다. 다음 조건이 모두 참인 같은 period snapshot만 전송할
수 있다.

1. 부가세 기간 요약이 존재한다.
2. 자료수집의 미수집·정규화 대기가 0건이다. 허용된 자료 부재 처리도 upstream에서 확정돼야 한다.
3. 자료대조원장의 증빙·소명·계정·제외·중복 등 미해결 blocker가 0건이다.
4. 부가세 공제 검토 pending이 0건이다.
5. 공제·불공제·안분과 과세·영세율·면세 사용자 판단이 모두 확정됐다.
6. 보류·전문가 확인·필수 증빙 미완료·안분 근거 미완료가 0건이다.
7. 확정 원장 provenance가 현재 fingerprint로 검증됐으며 rebuild_required가 아니다.
8. 사용자가 전송 직전 같은 fingerprint의 준비값·원본 범위·해결된 예외를 확인한다.

상류 도메인에서 이유·확인자·확인시각과 함께 해결 완료된 예외는 blocker로 다시 세지 않고
전송 내용에 포함한다. 위 조건 중 하나라도 실패하거나 확인 후 데이터가 바뀌면 CTA와 서버
전송을 모두 차단한다.

## 7. Initial Partner Pilot

초기 제품 검증은 실제 공개 마켓플레이스를 만들지 않고 다음 전제로 진행한다.

- 파트너 사무소 fixture 1곳
- `세무회계사무소 연결` 버튼 1개
- 사무소 정보·서비스 범위·자료 제공 범위 표시
- 연결 신청과 전체 세무자료 전달 동의를 분리
- 실제 고객 자료 전송·수임 계약·결제 없음
- 한 곳을 코드에 영구 고정하지 않고 향후 partner registry로 확장 가능한 개념 모델 사용

실제 출시 전에는 파트너 사무소 계약, 수임 성립 절차, 비용, 개인정보 처리 역할을 별도로
확정해야 한다.

A2A-1에서는 연결 신청·수임 승낙·자료 전달의 **순서와 게이트만** 정한다. 실제 동의 문구,
필수 고지와 개인정보 제공 범위는 A2A-5에서 처리자 역할·보관·철회 기준을 승인한 뒤 확정한다.

### 7.1 A2A-1 Connection Contract

초기 연결은 다음 흐름으로 확정한다.

1. 파일럿은 사전 제휴한 가상 파트너 사무소 1곳만 표시한다. 비교·추천·자동배정은 하지 않는다.
2. SemuAgent 사이드 네비게이션 하단에 `세무회계사무소 연결` 메뉴를 항상 표시한다.
3. 사업자가 `상담 신청`을 누르면 기존 사업자 정보로 신청서를 미리 채우고, 사업자가 연락처와
   요청 업무를 한 번 확인·보완한 뒤 전송한다.
4. JARYO-GIWA에 상담 요청 1건을 생성하고 담당자에게 이메일 알림을 보낸다. 이메일은 알림일
   뿐이며 상담 요청의 정본은 JARYO-GIWA다.
5. 상담 요청은 정식 수임 사업자 목록과 분리한다. 수임 계약이 완료되기 전에는 거래내역이나
   신고자료를 전달하지 않는다.
6. 사무소 담당자는 통화로 업무 범위와 수임료를 협의하고 외부에서 수임 계약을 체결한다.
7. 담당자가 JARYO-GIWA에서 `수임 계약 확인 및 연결 승인`을 누르면 연결이 완료된다. 신청 정보를
   정식 사업자 정보로 승격하되 기존 동일 사업자가 있으면 새로 만들지 않고 연결한다.
8. 연결 완료만으로 신고자료를 전송하지 않는다. 세목·신고기간별 Ready 화면에서 사업자가 매번
   별도의 `사무소로 전송`을 승인해야 한다.

A2A 시스템은 수임료 금액, 계약 조건, 계약서, 결제를 저장하거나 집행하지 않는다. 다음 상태와
감사 정보만 관리한다.

| 상태·행위 | 주체 | 처리 |
|---|---|---|
| 상담 요청 | 사업자 | JARYO 상담 목록에 생성하고 이메일 알림 |
| 상담 중 | 사무소 | 통화·업무 범위·수임료 협의 진행 |
| 상담 신청 취소 | 사업자 | 계약 전 요청 종료, 정식 사업자로 승격하지 않음 |
| 상담 종료 | 사무소 | 종료 사유를 기록하고 정식 사업자로 승격하지 않음 |
| 수임 계약 확인 및 연결 승인 | 사무소 | 외부 계약 체결 사실만 확인하고 연결 완료 |
| 연결 해지 | 사업자 | 신규 자료 전송 차단, 종료 사유·시각 기록 |
| 수임 종료 | 사무소 | 신규 자료 전송 차단, 종료 사유·시각 기록 |

상담·연결 종료 시 종료 주체, 종료 사유, 종료 시각을 기록한다. 기존 상담·전송 이력은 A2A-5의
보관 계약에 따라 감사 목적으로 보존한다. 연결 메뉴는 완료 후에도 네비게이션 하단에 유지해
연결된 사무소, 진행 상태, 취소·종료 이력을 확인하게 한다.

## 8. Receiver Review And Return

첫 전송 승인율을 높이고 사업자 재요청률을 낮추는 것이 핵심 운영 목표다. 사무소는 원본자료와
SemuAgent가 정리한 신고자료를 대조해, 기대한 Ready 품질로 전달됐는지 확인한다.

### 8.1 Direct Reception Contract

사업자가 세목·기간별 전송을 승인하면 SemuAgent는 구조화된 신고 준비자료와 그 판단에 사용된
원본자료를 JARYO-GIWA로 직접 전달한다. 담당자가 ZIP을 내려받아 다시 업로드하지 않는다.

JARYO-GIWA는 연결된 해당 사업자 화면에 다음 자료를 자동 등록한다.

- 원본 파일과 source reference
- 정규화된 통장·카드·세금계산서·현금영수증·급여 자료
- 증빙 연결, 계정 분류, 사용자 결정과 해결된 예외
- 세목·신고기간별 신고 준비값과 Ready snapshot
- contract version, source fingerprint, 파일 hash와 전송 감사 정보

수신 직후 상태는 `SemuAgent 수신 · 검토 대기`다. 수신만으로 JARYO-GIWA의 확정 장부나 신고
정본을 자동 변경하지 않는다. 담당자가 원본과 정리자료를 대조하고 `수신 승인`한 뒤 사무소
업무자료로 승격한다. 두 제품은 DB를 직접 공유하지 않고 versioned API·idempotency key·hash
검증으로 연결한다.

### 8.2 Receipt Acceptance And Canonical Boundary

`수신 승인`은 **해당 전송본을 사무소 업무의 입력자료로 사용해도 된다는 승인**이다. 장부나
신고값을 최종 확정했다는 의미가 아니다.

1. 검토 대기 중인 원본·정리자료·Ready snapshot은 하나의 변경 불가능한 수신 snapshot으로
   유지한다.
2. 담당자가 원본 범위·정리 결과·fingerprint·hash를 확인하고 수신 승인하면 상태를
   `승인된 수신자료`로 바꾼다.
3. 승인된 수신자료는 **수신 source의 정본**이지만 JARYO-GIWA의 확정 장부·신고 정본은 아니다.
4. 동일 세목·기간에서 가장 최근에 수신 승인한 SemuAgent snapshot을 현재 수신자료 정본으로
   사용한다.
5. JARYO-GIWA 업무 초안은 승인된 수신자료를 입력으로 사용할 수 있지만, 확정 장부·신고값
   승격은 기존 JARYO-GIWA의 별도 검토·확정 절차를 거친다.
6. v1은 같은 목적의 독립된 JARYO 자료와 일반적인 우선순위 경쟁·병합을 전제하지 않는다.
   동일 재시도와 보완 재전송 version만 명시적으로 처리한다.
7. 수신 승인 감사에는 승인자·승인시각·transfer ID·snapshot version·fingerprint와 이전 version
   관계를 보존한다.

따라서 `수신 승인` 한 번으로 기존 JARYO-GIWA 데이터를 일괄 덮어쓰거나 신고 준비를 완료 상태로
변경하지 않는다.

### 8.3 Linked Business Mapping

자료 전송 대상은 전송 시점마다 상호명이나 사업자등록번호로 다시 찾지 않는다. 수임 연결 승인
시점에 확정한 **고유 연결 ID**를 라우팅 정본으로 사용한다.

1. 사무소 담당자가 JARYO-GIWA에서 수임 연결을 승인할 때 기존 사업자를 선택하거나 새 사업자를
   생성한다.
2. 기존 동일 사업자가 있으면 반드시 그 사업자에 연결하고 중복 사업자를 만들지 않는다.
3. 연결 승인 결과로 SemuAgent 사업자와 JARYO-GIWA 사업자를 잇는 고유 연결 ID를 생성한다.
4. 이후 모든 전송은 활성 연결 ID, 파트너 사무소, 사업자 범위가 모두 일치할 때만 허용한다.
5. 상호명·사업자등록번호는 표시와 보조 검증에 사용할 수 있지만 전송 라우팅의 단독 정본으로
   사용하지 않는다.
6. JARYO-GIWA 사업자가 삭제·통합되거나 연결이 종료되면 해당 연결을 중지하고 재매핑·재승인
   전까지 전송을 차단한다.
7. 재매핑해도 과거 transfer와 당시 연결 대상 감사 이력은 변경하지 않는다.

양쪽 제품은 상대 제품의 내부 사업자 ID를 자체 FK처럼 직접 사용하거나 상대 DB를 조회하지
않는다. 연결 ID와 양쪽 reference의 정확한 구조는 versioned 공통 계약에서 확정한다.

### 8.4 Package-Level Acceptance

하나의 transfer는 일부 행만 수신 승인하는 `부분 수신` 상태를 만들지 않는다. 패키지 수준의
수신 결과는 다음 셋 중 하나다.

- `수신 승인`
- `보완 요청`
- `기술 오류`

사업자 사실이나 실제 원본이 부족해 패키지를 업무 입력자료로 사용할 수 없으면 전체 패키지를
`보완 요청`으로 돌린다. 사업자가 보완한 뒤 새 snapshot version의 전체 패키지를 재전송하며,
화면은 이전 version과 달라진 항목을 비교해 보여준다.

이 원칙은 `100건 중 95건 승인·5건 반송`처럼 여러 정본 version이 동시에 생기는 것을 막고,
재전송 중복과 누락을 줄인다.

### 8.5 Retry And Snapshot Version

v1에서 같은 목적의 자료가 다시 들어오는 경우는 **기술 재시도**와 **보완 후 새 version 전송**으로
한정해 처리한다.

1. 동일 transfer ID·contract version·fingerprint의 재시도는 새 수신 건을 만들지 않고 기존 수신
   결과를 그대로 반환한다.
2. 동일 transfer ID인데 fingerprint가 다르면 정상 재시도로 보지 않고 기술 오류로 거부한다.
3. 보완 후 전송은 이전 transfer를 참조하는 새 transfer ID와 새 snapshot version으로 등록하며,
   최초 상태는 다시 `SemuAgent 수신 · 검토 대기`다.
4. 새 version이 검토 대기인 동안 기존 승인 version을 임의로 교체하지 않는다.
5. 담당자가 새 version을 수신 승인하면 이전 승인 version은 `대체됨`으로 보존하고 새 version을
   현재 수신자료 정본으로 사용한다.
6. 이전 전송·보완 요청·승인·대체 이력은 삭제하거나 재작성하지 않는다.

이전 version을 기준으로 이미 진행한 사무소 작업을 새 version과 어떻게 동기화할지는 `A2A-4`에서
확정한다.

### 8.6 Review Outcome Boundary

수신 검토에서 발견한 문제는 다음 세 가지로만 구분한다.

1. **세무 판단 문제**: 받은 자료만으로 세무대리인이 판단할 수 있으면 사무소에서 처리한다.
2. **자료·사실 부족 문제**: 사업자만 알 수 있는 사실이나 실제 원본이 필요하면 SemuAgent로
   보완 요청한다.
3. **전송 시스템 문제**: 파일 손상, schema·hash·연결 ID 오류는 사업자 보완과 섞지 않고
   기술 오류로 처리한다.

사무소가 전문 범위에서 판단할 수 있는 문제를 편의상 사업자에게 되돌리지 않으며, 자료가 없는
상태에서 사실관계를 추정해 확정하지도 않는다.

### 8.7 Office Adjustment Feedback

SemuAgent Ready 자료는 사무소가 그대로 수신 승인하는 것을 기본으로 한다. 사무소 변경은 정상
반복 작업이 아니라 예외이며, 변경할 때는 다음 구조화 정보를 반드시 남긴다.

- 변경 대상과 변경 전·후 값
- 변경 사유 유형: `새로운 자료·사실`, `세무 전문가 판단`, `SemuAgent 판단 오류`,
  `신고 과정 수정`
- 실제 근거자료 또는 판단 근거 reference
- 처리 담당자와 처리시각

JARYO-GIWA AI는 위 구조화 정보와 실제 근거만 사용해 사업자가 이해하기 쉬운 변경사유 문장
초안을 작성할 수 있다. AI는 존재하지 않는 사실·증빙·법령 근거를 만들지 않으며, 담당자가
초안을 검토·수정하고 명시적으로 승인하기 전에는 SemuAgent로 보내지 않는다.

SemuAgent에는 담당자가 최종 승인한 설명문과 함께 구조화된 변경 전·후 값, 사유 유형, 근거
reference, 담당자·시각을 전달한다. `SemuAgent 판단 오류`는 Ready false positive와 품질 개선
지표에 연결한다.

### 8.8 SemuAgent Office Adjustment View

JARYO-GIWA 담당자가 최종 승인해 보낸 변경 결과는 SemuAgent의 기존 원본자료와 최초 판단을
덮어쓰지 않는다. 별도의 변경 불가능한 `세무사무소 확정 조정` 기록으로 저장한다.

- SemuAgent의 사무소 연결 전송 상세와 해당 세목 신고 진행상태·결과 화면에서 전송값과 사무소
  조정값을 함께 표시한다.
- 해당 신고 이벤트에서 최종 신고에 사용할 값은 사무소 조정값으로 표시하고 변경 이유·근거·
  담당자·처리시각을 함께 보여준다.
- 사업자의 추가 적용 승인을 기다리지 않고 반영하되, 변경 알림과 `사무소에 문의` 경로를
  제공한다.
- 사무소 조정을 다른 신고기간이나 향후 판단 규칙에 자동 확장하지 않는다.
- 사무소가 결과를 다시 수정하면 기존 기록을 삭제하지 않고 새 adjustment version으로 대체
  이력을 남긴다.

정확한 화면 위치와 정보 밀도는 A2A-7 양쪽 HTML Preview에서 확정한다.

### 8.9 Structured Supplement Request As Exception Recovery

보완 요청은 정상적인 사무소 반복 업무가 아니다. SemuAgent Ready gate가 미확정 자료를 막는 것이
기본이며, 보완 요청은 다음 예외에서만 사용한다.

- 전송 후 취소·환불 등 새로운 사실이 발생한 경우
- SemuAgent Ready 검사가 필요한 사실·자료를 놓친 경우

JARYO-GIWA 보완 요청은 일반 메일이나 자유문장만 보내지 않고 다음 구조화 정보를 포함한다.

- 대상 transfer·snapshot version
- 대상 원본·거래·판단 행 reference
- 예외 원인: `전송 후 새 사실` 또는 `SemuAgent Ready 누락`
- 사업자에게 묻는 내용과 필요한 자료
- 보완 요청 담당자·요청시각

SemuAgent는 해당 자료의 구조화 화면으로 연결되는 작업을 만들고 관련 신고 이벤트의 Ready를
다시 잠근다. 사업자는 SemuAgent에서 답변·자료를 보완하며, 이메일은 알림만 담당한다. 보완 후
전체 Ready gate를 다시 통과하고 사업자가 새 snapshot을 확인·재전송해야 한다.

`SemuAgent Ready 누락`은 Ready false positive로 집계해 같은 문제가 반복되지 않도록 gate·판단
규칙 개선에 연결한다.

### 8.10 Technical Retry

네트워크 지연이나 서버 일시 장애처럼 전송 내용과 무관한 기술 오류는 시스템이 자동 재시도한다.

1. 재시도는 최초 사용자가 승인한 동일 snapshot version·fingerprint·source scope만 사용한다.
2. 동일 idempotency key로 재시도해 JARYO-GIWA에 중복 수신 건을 만들지 않는다.
3. 내용이 변하지 않았으면 사업자에게 전송 승인을 다시 요구하지 않는다.
4. 제한된 자동 재시도 후에도 실패하면 SemuAgent에 `전송 실패` 상태와 명시적 `다시 전송`
   액션을 표시한다.
5. 재시도 전에 원본·판단·Ready fingerprint가 바뀌면 자동 재시도를 중단하고 전체 Ready 재검사와
   사업자 재확인을 요구한다.
6. 자동·수동 재시도 횟수, 시각, 결과와 최종 오류 코드를 감사 이력에 보존한다.

정확한 재시도 횟수·간격·backoff·timeout은 versioned 기술 계약에서 확정한다.

### 8.11 Minimal Filing Completion Signal

v1은 세무회계사무소의 현재 실무를 확인하지 않은 상태에서 접수번호·납부/환급세액·접수증 PDF
등록을 새 업무로 요구하지 않는다.

1. 실제 신고를 마친 JARYO-GIWA 담당자가 해당 신고 이벤트를 `신고 완료`로 처리한다.
2. SemuAgent에는 `세무사무소 신고 완료` 상태를 표시한다.
3. 세목·신고기간·처리 담당자·완료시각은 완료 처리 시 자동 감사 기록한다.
4. 사업자의 추가 승인을 기다리지 않고 결과 상태를 동기화한다.
5. 접수번호·최종 세액·접수증 PDF 등 추가 결과는 파일럿 사무소의 실제 신고 후 관리 방식을
   확인한 뒤 별도 질의응답으로 확장한다.

목표는 사무소 담당자에게 근거 없이 새로운 문서 등록 업무를 부과하지 않으면서 사업자에게 신고
완료 여부를 전달하는 것이다.

### 8.12 Privacy Handling At Engagement

개인정보 처리 관계와 필요한 계약·고지는 **세무회계사무소 수임계약 단계**에서 정리한다.
SemuAgent의 매 신고자료 전송마다 별도의 개인정보 동의 절차를 반복하지 않는다.

1. 상담 신청 전에는 상담에 필요한 최소 사업자·연락처 정보만 사무소에 전달하고 전달 사실을
   명확히 안내한다.
2. 수임계약과 연결 승인 전에는 거래·증빙·급여·신고자료를 보내지 않는다.
3. 수임계약 단계에서 개인정보 처리 역할, 업무 범위, 보호조치와 필요한 고지·계약을 정리한다.
4. 연결 완료 후 세목·기간별 `세무사무소로 전송`은 개인정보 동의 버튼이 아니라 이번 전송의
   수신 사무소·자료 범위를 확인하는 운영 승인이다.
5. 개인정보 처리의 정확한 법적 역할과 문구는 출시 전 개인정보·법무 검토를 통과해야 한다.

이 결정은 불필요한 반복 동의를 만들지 않으면서, 연결되지 않은 사무소로 자료가 자동 전달되는
것을 막는다.

### 8.13 Connection Termination And Retention

사무소 연결이 종료되면 미래의 자료 흐름은 즉시 끊되, 이미 전달된 신고자료를 자동 삭제하지
않는다.

1. 연결 종료 즉시 해당 연결 ID를 비활성화하고 신규·재전송을 차단한다.
2. SemuAgent에는 `연결 종료` 상태를 표시하고 사무소는 이후 SemuAgent 새 자료에 접근할 수 없다.
3. JARYO-GIWA의 기존 수신자료는 종료된 사업자 업무자료로 분리해 수임계약과 법적 의무에 따른
   기간 동안 보관한다.
4. 사업자가 삭제를 요청하면 법적으로 보관할 필요가 없는 자료만 정해진 절차로 삭제한다.
5. transfer ID·세목·기간·전송자·수신자·시각·상태 변경 같은 최소 감사 이력은 별도 보존한다.
6. 정확한 보관기간, 삭제 범위, 접근권한 축소 방식은 출시 전 법무 검토와 파일럿 사무소 실무
   확인 후 확정한다.

연결 종료는 미래 전송 권한을 끊는 동작이며 과거 신고 사실과 감사 이력을 소급해 없애는 동작이
아니다.

### 8.14 Active Filing Blocks Connection Termination

수임계약 아래 이미 전송한 신고 이벤트가 검토 대기·검토 중·보완 중 등 미완료 상태라면 일반
사용자 UI에서 연결을 종료할 수 없다.

1. SemuAgent와 JARYO-GIWA 양쪽에 `진행 중 신고를 완료한 후 연결을 종료할 수 있습니다`를
   표시한다.
2. v1에는 사무소가 정상 수임 업무를 임의로 중단하는 `신고 업무 중단` 액션을 만들지 않는다.
3. 해당 신고가 `신고 완료`된 뒤 외부 수임계약 종료가 확인되면 연결을 종료한다.
4. 분쟁, 계약 위반, 사무소 폐업처럼 정상 흐름으로 처리할 수 없는 경우는 운영자 수동 처리와
   감사 기록으로 분리하며 일반 종료 기능에 섞지 않는다.

제품은 수임계약을 맺은 사무소가 받은 신고를 완료하는 정상 흐름을 기본으로 한다.

### 8.15 Pilot Economics Deferred

SemuAgent와 파트너 사무소 사이의 소개비·중개수수료·수익배분 정책은 아직 결정하지 않는다.
v1 제품은 다음 경계를 지킨다.

- 사업자와 사무소의 수임료 상담·계약·결제를 앱에서 처리하지 않는다.
- SemuAgent는 상담 신청, 연결 상태, Ready 자료 전달과 처리 상태 확인만 담당한다.
- 초기 파트너 사무소와의 파일럿 비용·사업 계약은 제품 밖에서 별도로 관리한다.
- 사무소 비교·가격 비교·자동 배정·소개비 계산 기능을 만들지 않는다.
- A2A가 사무소 자료 정리·재입력 시간을 실제로 줄이는지 검증한 뒤 별도 사업 정책으로
  수익모델을 결정한다.

정책 미결정을 무료 제공이나 특정 수수료 모델 확정으로 해석하지 않는다.

### 8.16 Partner Response Responsibility

상담 신청에 얼마나 빨리 응답할지는 신규 고객을 수임하려는 파트너 사무소의 영업 책임이다.
SemuAgent가 응답 기한이나 독촉 절차를 운영하지 않는다.

- 상담 신청을 JARYO-GIWA에 즉시 등록하고 담당자에게 최초 이메일 알림을 보낸다.
- 사업자에게 `상담 신청 완료`와 이후 사무소가 변경한 상태를 표시한다.
- 사무소가 응답하지 않으면 사업자는 기존 `신청 취소`를 사용할 수 있다.
- v1에는 응답 SLA, 반복 자동 독촉, SemuAgent 운영자 개입, 다른 사무소 자동 재배정을 넣지 않는다.
- 사무소 응답시간은 파일럿 효과 지표로 측정할 수 있지만 제품이 강제하지 않는다.

초기 파트너는 사전 계약한 한 곳이므로 상담 경쟁·배분 로직을 만들지 않는다.

### 8.17 Separate Connection Management From Filing Transfer

SemuAgent의 `세무사무소 연결 관리`와 세목별 `신고자료 전송`은 서로 다른 화면 책임으로
분리한다.

1. 사이드 네비게이션 하단의 `세무사무소 연결` 화면은 상담 신청, 상담 상태, 수임계약 확인,
   연결 완료·종료, 파트너 사무소 정보와 전송 이력을 관리한다.
2. 각 세목 화면은 해당 세목·신고기간의 Ready가 충족됐을 때만 `세무사무소로 보내기`를
   제공한다. 첫 적용 세목은 부가세다.
3. 세목별 전송 화면은 수신 사무소, 신고기간, 준비값, 원본 범위, 해결된 예외, snapshot version과
   Ready 상태를 확인한 뒤 명시적으로 전송하는 데 집중한다.
4. 부가세 화면의 `세무사무소로 보내기`는 즉시 전송하거나 작은 모달을 열지 않는다. 별도의
   전송 확인 화면으로 이동해 위 범위를 충분히 검토한 뒤 `세무사무소로 전송`(§5.3)으로
   최종 승인한다.
5. 연결 화면은 장기적인 수임 관계를, 세목 화면은 반복되는 개별 신고 이벤트를 관리한다. 같은
   상태를 중복 편집하지 않고 필요한 경우 서로의 상세 화면으로 연결한다.
6. JARYO-GIWA에는 고객사 전체를 가로지르는 신규 `SemuAgent 수신` 업무 목록을 만든다. 새
   전송의 고객사·세목·기간·수신 상태·담당자를 한눈에 확인하는 업무 입구다.
7. 수신 목록의 항목을 열면 연결된 고객사 아래의 신규 `SemuAgent 신고자료 상세`로 이동한다.
   기존 고객사 상세에는 이 영역으로 들어가는 탭 또는 최근 수신 건 링크를 제공한다.
8. 수신 목록과 고객사별 상세는 같은 transfer record를 조회하며 데이터를 복제하지 않는다.
   수신 승인·보완 요청·기술 오류 같은 실제 검토 액션은 고객사별 상세에서 처리한다.
9. 기존 `자료 검토`는 파일 제출·누락 검토 책임을 유지한다. A2A 신고 snapshot·version·사무소
   조정 업무를 기존 화면에 억지로 합치지 않고 새 수신 작업공간으로 분리한다.
10. 고객사별 신고자료 상세의 첫 화면은 고객사·세목·기간, Ready, 신고 준비값, 원본 종류·건수,
    해결된 예외·주의사항과 핵심 처리 액션만 보여준다. 개별 원본·거래 행은 탭이나 펼쳐보기에서
    확인하며 첫 화면에 전체 데이터를 쏟아놓지 않는다.
11. SemuAgent 전송 확인 화면과 JARYO 신고자료 상세는 같은 versioned 표시 계약을 사용하는
    공통 읽기 전용 snapshot viewer로 금액·원본 범위·예외를 동일하게 표시한다.
12. SemuAgent의 편집·AI 재판단·Ready 계산과 JARYO의 수신 승인·보완 요청·기술 오류·사무소
    조정은 공통 viewer 바깥의 각 제품 전용 액션으로 분리한다.
13. 공통 viewer는 iframe, 상대 제품 DB 직접 조회나 전체 runtime page 복사로 구현하지 않는다.
    versioned snapshot schema와 제품별 adapter를 사용하며 정확한 공유 모듈 경계는 기술 계약에서
    확정한다.
14. 사업자가 전송을 승인하면 해당 snapshot에 실제 사용된 원본과 정리자료를 JARYO-GIWA 전용
    저장소로 서비스 간 직접 복사한다. 사용자가 ZIP을 내려받아 다시 업로드하는 절차는 만들지
    않는다.
15. JARYO-GIWA는 원본별 hash와 source metadata를 저장하고 수신 전에 검증한다. 원본 누락·hash
    불일치·복사 실패가 있으면 검토 대기가 아닌 기술 오류로 처리한다.
16. 수신 후 공통 viewer와 사무소 검토는 JARYO-GIWA에 보관된 고정 사본을 읽는다. SemuAgent
    파일의 실시간 링크나 연결 종료 뒤 접근 가능한 hot link에 의존하지 않는다.
17. 원본의 정확한 암호화·저장 구조·대용량 전송 재시도는 versioned 기술 계약에서 확정한다.
    이미 수신 승인한 사본의 보관·삭제 경계는 §8.13의 수임계약·법적 의무 원칙을 따른다.
18. 사업자는 SemuAgent에서만, 사무소 담당자는 JARYO-GIWA에서만 각자 로그인한다. 상대 제품에
    사용자 비밀번호·로그인 세션·브라우저 cookie를 전달하거나 공유하지 않는다.
19. SemuAgent 서버는 사업자의 전송 권한·Ready·현재 fingerprint를 먼저 검증한 뒤, 파트너별
    서버 간 인증 정보와 요청 서명으로 JARYO-GIWA 수신 API를 호출한다.
20. JARYO-GIWA는 파트너 인증, 요청 서명, contract version, 활성 연결 ID, 수신 tenant와
    대상 사업자 매핑을 모두 확인한 요청만 수신한다. 어느 하나라도 맞지 않으면 fail-closed로
    거부한다.
21. 파트너 연결·계약을 종료하거나 인증 정보를 폐기하면 이후 전송 권한도 즉시 차단한다. 정확한
    credential 형식·회전·만료·서명 알고리즘은 versioned 기술 계약에서 확정한다.
22. 공통 전송 계약은 세목과 무관한 envelope와 세목별 payload를 분리한다. envelope에는
    transfer ID, 연결 ID, 사업자 reference, 세목, 신고기간, snapshot version, fingerprint,
    hash, 전송시각을 포함한다.
23. 첫 payload는 부가세 준비값·실제 사용 원본 범위·해결된 예외를 담는다. 원천세·지급명세서 등
    후속 세목은 envelope를 바꾸지 않고 각 세목 payload contract를 추가한다.
24. JARYO-GIWA는 원본의 versioned payload를 보존하고 공통 viewer의 제품별 adapter가 이를
    표시한다. 세목별 필드를 공통 envelope에 하드코딩하지 않는다.
25. 정확한 Zod schema, API path, error code와 compatibility matrix는 A2A-7 기술 명세에서
    확정한다.
26. 정확한 정보 밀도·버튼 위치·모바일 동선은 A2A-7 HTML Preview에서 오너 확인 후 확정한다.

수신 후 오류가 발견되면 다음처럼 분리한다.

| 수신 결과 | 처리 원칙 |
|---|---|
| 수신 승인 | 원본과 정리 자료가 일치해 `승인된 수신자료`로 승격. 장부·신고 정본 확정은 별도 절차 |
| 보완 요청 | 사업자만 확인할 수 있는 사실 또는 실제 자료가 필요해 SemuAgent로 반환 |
| 기술 오류 | 파일 손상, schema·version·hash·mapping 오류를 별도 처리하고 재전송 요청 |

세무대리인이 전문 범위에서 판단할 수 있는 사항은 보완 요청으로 돌리지 않고 사무소에서 처리한
뒤 근거와 수정 결과를 기록한다.

사무소 수정 결과는 §8.7~§8.8의 별도 adjustment·변경사유·최종 신고값 표시 계약을 따른다.

## 9. Delivery Evolution

| 단계 | 내용 | 현재 상태 |
|---|---|---|
| **Fallback** | SemuAgent ZIP Export, 비연동·장애 상황에서 수동 전달 | JC-034 문서 완료, 구현 보류 |
| **A2A v1** | 연결된 사업자의 구조화 자료·원본을 JARYO 사업자 화면에 직접 등록, 최초 검토 대기 | 제품·수신 경계 승인, UI·기술 계약 대기 |
| **A2A v2** | 수신 승인·보완 요청·기술 오류·재전송 상태 동기화 | 제품 계약 승인, UI·기술 계약 대기 |
| **A2A v3** | 사무소 수정 결과와 신고 완료 결과를 양방향 동기화 | 최소 제품 계약 승인, 확장 결과 항목은 파일럿 확인 대기 |

모든 단계에서 사업자와 사무소 담당자의 명시적 확인을 유지한다.

## 13. Governance

### 13.1 Current document ownership

- 본 문서를 두 제품의 **임시 공통 정본**으로 둔다.
- SemuAgent는 생산자·사업자 경험 문서를 관리한다.
- JARYO-GIWA는 수신자·사무소 경험 문서를 관리하고 본 문서를 참조한다.
- 같은 공통 계약을 양쪽 저장소에 복사하지 않는다.
- JARYO-GIWA Concept 32 / Spec 42가 본 문서와 충돌하면 Owner Q&A 결정, SemuAgent JC-044
  직접 연동 계약, JC-034 ZIP fallback 계약을 함께 확인해 수정한다.

### 13.2 Before v2 implementation

v2 API·상태 동기화 착수 전에는 공통 계약을 별도 공유 저장소 또는 versioned schema package로
분리한다. `contractVersion`, changelog, compatibility matrix를 양쪽 제품이 함께 사용한다.

## 14. Non-Goals At This Stage

- 공개 세무사무소 마켓플레이스
- 무작위 추천·자동 배정
- 실제 파트너 계약 없이 고객자료 전송
- 처리 방향이 확정되지 않은 자료 또는 치명적 무결성 오류가 있는 패키지 전송
- 수신 즉시 JARYO 확정 장부·신고 정본으로 자동 승격
- 두 제품의 DB 직접 연결·공유
- 자동 신고·자동 납부
- 홈택스·위택스·검정 SW 자격증명 저장
- API·DB·UI 구현 착수
