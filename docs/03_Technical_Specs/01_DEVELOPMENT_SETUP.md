## Reuse Guard

Before reusing a JARYO-GIWA module:

1. Identify whether it is product-neutral, accounting-firm-specific, or company
   self-use-specific.
2. Keep product-neutral logic where possible.
3. Rename or adapt accounting-firm-specific entities before exposing them in
   UI/API.
4. Preserve tests when the business rule is unchanged.
5. Add new tests when the responsibility boundary changes.

network access, run `npm install --package-lock-only --ignore-scripts` in an
approved network environment.

## Vercel Deployment Cost Guard

The Vercel project uses a **fixed Standard** build machine. Do not switch it to
Elastic, Enhanced, or Turbo without an explicit cost review.

Automatic Git deployments are intentionally limited in `vercel.json`:

- `main` creates the Production deployment.
- `preview` creates an intentional Preview deployment for browser QA.
- feature, fix, and docs branches do not create Vercel deployments on every
  push.

When a PR needs browser verification, update the dedicated `preview` branch
only after the change is ready for visual QA. TypeScript, unit tests, lint, and
local browser checks remain the default verification path for intermediate
commits. Documentation-only changes must not trigger remote builds.

This guard was added after 468 deployments were created between 2026-07-03 and
2026-07-12 (287 Preview and 181 Production), producing 11,320 billable Build CPU
Minutes. Keep Production merges intentional and avoid splitting one reviewable
change into multiple deployment-triggering pushes.
