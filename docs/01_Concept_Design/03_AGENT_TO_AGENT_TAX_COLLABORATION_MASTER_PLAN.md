# Agent-to-Agent Tax Collaboration Master Plan

## 1. Purpose

이 문서는 사업자 측 **SemuAgent**와 세무회계사무소 측 **JARYO-GIWA**를 연결하는
Agent-to-Agent 세무 협업의 상위 제품 계획을 정의한다.

이 문서는 구현 명세가 아니다. 현재까지 사용자와 합의한 제품 원칙을 기록하고,
남은 결정을 `A2A-0`부터 `A2A-7`까지 질의응답 단위로 나누어 확정하기 위한
Master Plan이다.

## 2. North Star

```text
사업자: SemuAgent로 자료 수집·대조·세무 판단·신고 준비
  -> 기본 경로: 사업자가 홈택스·위택스에서 직접 신고
  -> 선택 경로: 준비 완료 자료를 계약된 파트너 사무소에 전달
세무회계사무소: JARYO-GIWA로 정제된 자료를 수신·검토
  -> 세무대리인이 최종 판단
  -> 검정 SW로 신고
```

SemuAgent는 사업자가 직접 신고할 수 있도록 자료를 정리하고 안내하는 것이 기본이다.
직접 신고가 번거로운 사업자에게는 선택적으로 `세무회계사무소 연결`을 제공한다.

JARYO-GIWA를 사용하는 사무소는 이미 수집·대조·분류·확정된 자료를 받는다. 따라서
자료 요청, 누락 확인, 형식 통일, 재입력, 반복 문의에 쓰는 시간을 줄이고 세무 판단과
최종 신고에 집중할 수 있다.

## 3. Product Value

### 3.1 사업자

- 직접 신고를 기본 선택으로 유지한다.
- 원할 때만 계약된 세무회계사무소에 준비 완료 자료를 전달한다.
- 사무소에 맡기더라도 같은 자료를 다시 모으거나 설명하지 않는다.
- 전달 범위와 수신자를 확인한 뒤 직접 전송한다.

### 3.2 세무회계사무소

- 원본자료와 SemuAgent가 정리한 신고 준비 자료를 함께 받아 대조한다.
- 자료 수집·정리·재입력 업무를 크게 줄인다.
- 사업자 사실 확인이 필요한 예외만 구조화해 되돌려보낸다.
- 더 적은 반복 작업으로 더 많은 고객을 안정적으로 처리할 수 있다.

### 3.3 두 제품

- SemuAgent는 사업자의 세무 준비 에이전트다.
- JARYO-GIWA는 세무회계사무소의 업무 수행 에이전트다.
- 두 에이전트는 사람의 승인 아래 검증된 자료, 상태, 보완 요청과 처리 결과를 주고받는다.
- 사업자에게는 신고 선택권을, 사무소에는 신규 수임 기회와 낮은 처리비용을 제공한다.

## 4. Approved Product Principles

현재 질의응답으로 다음 원칙을 **제품 방향 수준에서** 승인했다. 상태·필드·문구·예외 처리의
세부 계약까지 승인된 것은 아니며, 단위별 상세 승인 상태는 §10을 따른다.

1. **직접 신고가 기본이다.** 세무회계사무소 연결은 선택 기능이다.
2. **초기 파일럿은 가상 파트너 사무소 1곳**으로 설계한다.
3. 실제 연결 출시 전에는 JARYO-GIWA 사용 사무소와 별도 사업 계약이 선행되어야 한다.
4. 신고 일정과 대상 기간은 법정 일정에 따라 SemuAgent가 결정한다.
5. SemuAgent는 해당 세목·기간 자료를 자동으로 준비하지만 **자동 전송하지 않는다.**
6. 사업자가 수신자, 세목, 기간, 포함 자료와 동의 범위를 확인하고 전송 버튼을 눌러야 한다.
7. 처리 방향이 정해지지 않은 미확정 항목과 치명적 무결성 오류가 없는 **Ready 패키지만**
   사무소에 전송한다.
8. JARYO-GIWA는 Ready 패키지를 최종 검토하며 자동으로 신고 확정하지 않는다.
9. 세무대리인이 전문 범위에서 결정 가능한 사항은 사무소가 처리하고 근거를 남긴다.
10. 사업자만 확인할 수 있는 사실이 필요하면 SemuAgent로 구조화된 수정 요청을 반환한다.
11. 홈택스 제출은 사무소와 검정 SW가 수행한다. 두 에이전트 모두 자동 신고하지 않는다.
12. 연결된 사무소 전송은 **SemuAgent에서 JARYO-GIWA로 직접 연동**하는 것을 기본으로 한다.
13. JARYO-GIWA에 수신된 자료는 해당 사업자 화면에 자동 등록하되 최초 상태는 **검토 대기**다.
14. ZIP 다운로드·수동 업로드는 비연동·장애 상황의 보조 경로이며 A2A 주 경로가 아니다.

## 5. Ready Handoff Principle

Agent-to-Agent 연결의 핵심은 `보냈다`가 아니라 `다시 정리할 필요가 없는 상태로 보냈다`다.

Ready는 사업장 전체가 아니라 **세목·신고기간별**로 판정한다. 한 세목의 미확정 항목이
다른 세목의 Ready 패키지 전송을 막지 않는다.

Ready는 최소한 다음 조건을 만족해야 한다.

- 대상 신고 세목과 기간이 확정됨
- 필수 원천자료 수집 또는 자료 부재에 대한 처리 방향 확정
- 누락·중복·금액 불일치 해결 또는 허용된 예외 처리 완료
- 증빙 연결 완료
- 계정항목과 거래 목적 확정
- 세목별 공제·과세유형 등 사용자 확인 완료
- stale 계산, 합계 불일치, 손상된 데이터, 필수 주체 정보 누락 등 치명적 무결성 오류 0건
- 사업자가 전달 내용을 최종 확인함

자료가 없다는 사실만으로 Ready를 영구 차단하지 않는다. 사용자가 `증빙 없음`, `공제 제외`,
`이번 신고에서 제외`처럼 허용된 처리 방향을 명시적으로 확정하면 해당 항목은 해결된 예외로
간주한다. 이 결정은 이유·확인자·확인시각과 함께 패키지에 포함한다.

반대로 매출 누락 가능성, 산술 불일치, stale 계산, 데이터 손상처럼 신고 결과를 신뢰할 수 없게
만드는 문제는 사용자의 위험 수용만으로 해제할 수 없다. 전송 화면과 서버가 같은 Ready 계약을
재검증한다.

세목별 정확한 Ready 조건은 `A2A-2` 질의응답에서 별도 완료 계약으로 세분화한다.

### 5.1 Transmission Unit

A2A의 정본 전송 단위는 **사업자 1곳 × 신고 세목 1개 × 신고기간 1개 × snapshot version 1개**다.
같은 마감일의 여러 세목을 화면에서 함께 선택할 수는 있지만, 내부적으로는 세목별 독립 전송
이벤트를 생성한다.

각 전송 이벤트는 독립된 다음 정보를 가진다.

- transfer ID와 contract version
- 신고 세목·신고기간
- Ready snapshot·source fingerprint
- 사용자 전송 승인
- JARYO 수신·검토·보완·기술 오류 상태
- 재전송 version과 감사 이력

한 세목의 blocker·보완 요청·기술 오류가 같은 날 선택한 다른 세목의 수신 승인에 영향을 주지
않는다. 여러 세목을 하나의 acceptance 결과로 묶지 않는다.

### 5.2 Source Scope

각 전송에는 해당 세목·신고기간의 Ready 판단과 신고 준비값 생성에 **실제로 사용된 자료만**
포함한다.

포함 범위:

- 신고 준비 행이 참조하는 세금계산서·카드·현금영수증·통장·급여 등 원본자료
- 해당 신고 이벤트에 포함된 정규화 행
- 증빙 연결, 계정 분류, 공제·과세유형과 사용자 확정
- 해결된 예외와 사유·확인자·확인시각
- Ready snapshot, validation 결과, provenance·fingerprint·hash

제외 범위:

- 다른 신고 세목·다른 신고기간의 자료
- 회사의 전체 과거 거래·전체 문서 보관함
- 해당 snapshot이 참조하지 않는 초안·폐기·대체된 자료
- 홈택스·위택스·은행·카드 자격증명과 비밀정보

하나의 원본이 여러 신고 이벤트의 근거로 실제 사용되면 각 이벤트가 그 source reference를
가질 수 있다. 물리 파일은 source ID·hash로 중복 전송을 줄일 수 있지만, 권한과 감사 범위는
각 전송 이벤트별로 독립 유지한다.

### 5.3 Pre-Send User Confirmation

사업자는 전송 직전에 별도 확인 화면에서 최소한 다음 항목을 확인한다.

- 수신 세무회계사무소와 현재 연결 상태
- 대상 사업자, 신고 세목, 신고기간, snapshot version
- 신고 준비값 요약
- 자료유형별 포함 원본 건수와 펼쳐볼 수 있는 상세 원본 목록
- 해결된 예외와 각 처리 방향
- Ready 상태와 미해결 blocker 0건

최종 CTA는 `세무사무소로 전송`으로 표시한다. 사용자가 확인한 fingerprint와 서버의 현재
fingerprint가 다르거나 Ready가 해제되면 전송을 차단하고 최신 내용 재확인을 요구한다.
전송 감사 이력에는 사용자, 확인시각, 수신 사무소, 세목·기간, snapshot version, fingerprint,
포함 source scope를 남긴다.

이 확인은 현재 자료를 실제로 전송하는 **행위 승인**이다. 개인정보 처리 역할과 법적 자료전달
동의 문구는 A2A-5에서 별도로 확정하며, 그 결과가 이 화면의 동의 문구 선행조건이다.

### 5.4 Ready Acceptance Gap

SemuAgent가 Ready로 판정한 패키지를 JARYO-GIWA가 자료 누락이나 무결성 문제로 반송하면,
이를 정상 운영 예외로 취급하지 않는다. **생산자 Ready 기준과 수신자 acceptance 기준 사이의
계약 간극**으로 기록하고 원인을 분류해야 한다.

A2A-2와 A2A-3에서는 최소한 다음 반송 사유를 구분한다.

- SemuAgent Ready false positive
- schema·contract version 또는 hash 무결성 실패
- Ready 이후 원천자료 변경·stale 발생
- 사무소 검토에서 새로 드러난 사업자 사실 확인 필요

포괄적인 `자료 누락` 한 가지로 기록하지 않는다. Ready 패키지 반송은 Ready 완료 계약 개선과
첫 수신 승인율 지표에 반드시 되먹임한다.

## 6. Filing-Schedule-Driven Packages

패키지는 임의 월간 묶음이 아니라 법정 신고 일정의 세목·기간 단위로 준비한다. 화면의 일괄
선택은 편의 기능일 뿐이며, 전송·Ready·감사·반송 정본은 항상 독립 이벤트로 유지한다.

| 신고 이벤트 | 준비 대상 예시 | 정확한 계약 상태 |
|---|---|---|
| 원천세 | 해당 귀속·지급월 급여와 원천세 준비값 | A2A-2에서 확정 |
| 부가세 | 해당 과세기간 매출·매입·공제·과세유형 | **첫 구현·검증 세목 Owner Approved**, 세부 Ready 계약 대기 |
| 지급명세서 | 반기·연간 지급자료 | A2A-2에서 확정 |
| 연간 신고 | 종합소득세·법인세 대상 연도 자료 | 해당 기능 완료 후 확정 |

SemuAgent는 신고 일정에 맞춰 패키지를 미리 준비하고, Ready가 되기 전에는 전송 버튼을
활성화하지 않는다.

### 6.1 First Implementation Tax Type

A2A 직접 연동의 첫 구현·브라우저 E2E·파트너 fixture 검증 세목은 **부가세**다. 부가세는
세금계산서·카드·현금영수증·통장 원본, 자료대조, 공제·불공제·안분과 과세유형 사용자 결정,
Ready gate를 함께 사용하므로 두 제품의 연결 가치를 가장 넓게 검증할 수 있다.

부가세 E2E 완료 후 같은 versioned 전송·수신 계약을 원천세, 지급명세서와 다른 신고 이벤트로
확대한다. 이 결정은 최종 지원 범위를 부가세로 제한하지 않는다.

### 6.2 VAT Ready Contract

첫 부가세 A2A slice는 별도의 Ready 공식을 새로 만들지 않고 현재 VAT package gate의
`isReady`를 단일 원천으로 재사용한다. 다음 조건이 모두 참인 같은 period snapshot만 전송할
수 있다.

1. 부가세 기간 요약이 존재한다.
2. 자료수집의 미수집·정규화 대기가 0건이다. 허용된 자료 부재 처리도 upstream에서 확정돼야 한다.
3. 자료대조원장의 증빙·소명·계정·제외·중복 등 미해결 blocker가 0건이다.
4. 부가세 공제 검토 pending이 0건이다.
5. 공제·불공제·안분과 과세·영세율·면세 사용자 판단이 모두 확정됐다.
6. 보류·전문가 확인·필수 증빙 미완료·안분 근거 미완료가 0건이다.
7. 확정 원장 provenance가 현재 fingerprint로 검증됐으며 rebuild_required가 아니다.
8. 사용자가 전송 직전 같은 fingerprint의 준비값·원본 범위·해결된 예외를 확인한다.

상류 도메인에서 이유·확인자·확인시각과 함께 해결 완료된 예외는 blocker로 다시 세지 않고
전송 내용에 포함한다. 위 조건 중 하나라도 실패하거나 확인 후 데이터가 바뀌면 CTA와 서버
전송을 모두 차단한다.

## 7. Initial Partner Pilot

초기 제품 검증은 실제 공개 마켓플레이스를 만들지 않고 다음 전제로 진행한다.

- 파트너 사무소 fixture 1곳
- `세무회계사무소 연결` 버튼 1개
- 사무소 정보·서비스 범위·자료 제공 범위 표시
- 연결 신청과 전체 세무자료 전달 동의를 분리
- 실제 고객 자료 전송·수임 계약·결제 없음
- 한 곳을 코드에 영구 고정하지 않고 향후 partner registry로 확장 가능한 개념 모델 사용

실제 출시 전에는 파트너 사무소 계약, 수임 성립 절차, 비용, 개인정보 처리 역할을 별도로
확정해야 한다.

A2A-1에서는 연결 신청·수임 승낙·자료 전달의 **순서와 게이트만** 정한다. 실제 동의 문구,
필수 고지와 개인정보 제공 범위는 A2A-5에서 처리자 역할·보관·철회 기준을 승인한 뒤 확정한다.

### 7.1 A2A-1 Connection Contract

초기 연결은 다음 흐름으로 확정한다.

1. 파일럿은 사전 제휴한 가상 파트너 사무소 1곳만 표시한다. 비교·추천·자동배정은 하지 않는다.
2. SemuAgent 사이드 네비게이션 하단에 `세무회계사무소 연결` 메뉴를 항상 표시한다.
3. 사업자가 `상담 신청`을 누르면 기존 사업자 정보로 신청서를 미리 채우고, 사업자가 연락처와
   요청 업무를 한 번 확인·보완한 뒤 전송한다.
4. JARYO-GIWA에 상담 요청 1건을 생성하고 담당자에게 이메일 알림을 보낸다. 이메일은 알림일
   뿐이며 상담 요청의 정본은 JARYO-GIWA다.
5. 상담 요청은 정식 수임 사업자 목록과 분리한다. 수임 계약이 완료되기 전에는 거래내역이나
   신고자료를 전달하지 않는다.
6. 사무소 담당자는 통화로 업무 범위와 수임료를 협의하고 외부에서 수임 계약을 체결한다.
7. 담당자가 JARYO-GIWA에서 `수임 계약 확인 및 연결 승인`을 누르면 연결이 완료된다. 신청 정보를
   정식 사업자 정보로 승격하되 기존 동일 사업자가 있으면 새로 만들지 않고 연결한다.
8. 연결 완료만으로 신고자료를 전송하지 않는다. 세목·신고기간별 Ready 화면에서 사업자가 매번
   별도의 `사무소로 전송`을 승인해야 한다.

A2A 시스템은 수임료 금액, 계약 조건, 계약서, 결제를 저장하거나 집행하지 않는다. 다음 상태와
감사 정보만 관리한다.

| 상태·행위 | 주체 | 처리 |
|---|---|---|
| 상담 요청 | 사업자 | JARYO 상담 목록에 생성하고 이메일 알림 |
| 상담 중 | 사무소 | 통화·업무 범위·수임료 협의 진행 |
| 상담 신청 취소 | 사업자 | 계약 전 요청 종료, 정식 사업자로 승격하지 않음 |
| 상담 종료 | 사무소 | 종료 사유를 기록하고 정식 사업자로 승격하지 않음 |
| 수임 계약 확인 및 연결 승인 | 사무소 | 외부 계약 체결 사실만 확인하고 연결 완료 |
| 연결 해지 | 사업자 | 신규 자료 전송 차단, 종료 사유·시각 기록 |
| 수임 종료 | 사무소 | 신규 자료 전송 차단, 종료 사유·시각 기록 |

상담·연결 종료 시 종료 주체, 종료 사유, 종료 시각을 기록한다. 기존 상담·전송 이력은 A2A-5의
보관 계약에 따라 감사 목적으로 보존한다. 연결 메뉴는 완료 후에도 네비게이션 하단에 유지해
연결된 사무소, 진행 상태, 취소·종료 이력을 확인하게 한다.

## 8. Receiver Review And Return

첫 전송 승인율을 높이고 사업자 재요청률을 낮추는 것이 핵심 운영 목표다. 사무소는 원본자료와
SemuAgent가 정리한 신고자료를 대조해, 기대한 Ready 품질로 전달됐는지 확인한다.

### 8.1 Direct Reception Contract

사업자가 세목·기간별 전송을 승인하면 SemuAgent는 구조화된 신고 준비자료와 그 판단에 사용된
원본자료를 JARYO-GIWA로 직접 전달한다. 담당자가 ZIP을 내려받아 다시 업로드하지 않는다.

JARYO-GIWA는 연결된 해당 사업자 화면에 다음 자료를 자동 등록한다.

- 원본 파일과 source reference
- 정규화된 통장·카드·세금계산서·현금영수증·급여 자료
- 증빙 연결, 계정 분류, 사용자 결정과 해결된 예외
- 세목·신고기간별 신고 준비값과 Ready snapshot
- contract version, source fingerprint, 파일 hash와 전송 감사 정보

수신 직후 상태는 `SemuAgent 수신 · 검토 대기`다. 수신만으로 JARYO-GIWA의 확정 장부나 신고
정본을 자동 변경하지 않는다. 담당자가 원본과 정리자료를 대조하고 `수신 승인`한 뒤 사무소
업무자료로 승격한다. 두 제품은 DB를 직접 공유하지 않고 versioned API·idempotency key·hash
검증으로 연결한다.

수신 후 오류가 발견되면 다음처럼 분리한다.

| 수신 결과 | 처리 원칙 |
|---|---|
| 수신 승인 | 원본과 정리 자료가 일치하며 사무소 검토 시작 |
| 보완 요청 | 사업자만 확인할 수 있는 사실 또는 실제 자료가 필요해 SemuAgent로 반환 |
| 기술 오류 | 파일 손상, schema·version·hash·mapping 오류를 별도 처리하고 재전송 요청 |

세무대리인이 전문 범위에서 판단할 수 있는 사항은 보완 요청으로 돌리지 않고 사무소에서 처리한
뒤 근거와 수정 결과를 기록한다.

사무소 수정 결과를 SemuAgent에 어떤 수준으로 되돌릴지는 `A2A-4`의 미결정 사항이다.

## 13. Governance

### 13.1 Current document ownership

- 본 문서를 두 제품의 **임시 공통 정본**으로 둔다.
- SemuAgent는 생산자·사업자 경험 문서를 관리한다.
- JARYO-GIWA는 수신자·사무소 경험 문서를 관리하고 본 문서를 참조한다.
- 같은 공통 계약을 양쪽 저장소에 복사하지 않는다.
- JARYO-GIWA Concept 32 / Spec 42가 본 문서와 충돌하면 Owner Q&A 결정, SemuAgent JC-044
  직접 연동 계약, JC-034 ZIP fallback 계약을 함께 확인해 수정한다.

### 13.2 Before v2 implementation

v2 API·상태 동기화 착수 전에는 공통 계약을 별도 공유 저장소 또는 versioned schema package로
분리한다. `contractVersion`, changelog, compatibility matrix를 양쪽 제품이 함께 사용한다.

## 14. Non-Goals At This Stage

- 공개 세무사무소 마켓플레이스
- 무작위 추천·자동 배정
- 실제 파트너 계약 없이 고객자료 전송
- 처리 방향이 확정되지 않은 자료 또는 치명적 무결성 오류가 있는 패키지 전송
- 수신 즉시 JARYO 확정 장부·신고 정본으로 자동 승격
- 두 제품의 DB 직접 연결·공유
- 자동 신고·자동 납부
- 홈택스·위택스·검정 SW 자격증명 저장
- API·DB·UI 구현 착수
