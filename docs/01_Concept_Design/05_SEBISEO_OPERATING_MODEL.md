## 1. 문서 목적

이 문서는 SemuAgent의 제품 원칙인 **"대화로 시키고 이해하며, 표에서 검토하고 확정한다"**를
실제로 운영 가능한 제품 계약으로 구체화한다.

기존 [대화형 세무 작업공간 방향](./04_CONVERSATIONAL_TAX_WORKSPACE_PRODUCT_DIRECTION.md)이
세비서를 중심 진입점으로 두는 이유와 책임 경계를 정의한다면, 이 문서는 다음 질문에 답한다.

- 사용자의 한 문장이 어떤 업무로 해석되는가?
- 업무는 어떤 단계로 진행되고, 무엇 때문에 멈추는가?
- 세비서는 무엇을 직접 수행하고 무엇을 구조화 화면으로 넘기는가?
- `Ready`, `완료`, `제출됨`은 어떤 근거로 판정되는가?
- 실패하거나 중단된 업무를 어떻게 다시 시작하는가?
- 사용자의 확인과 변경 이력을 어떻게 보존하는가?

이 문서는 제품 운영모델을 정의한다. 상세 화면, DB 스키마, API, 큐와 실행 인프라는 이 문서를
입력으로 별도 UI 문서와 Pre-Code Technical Brief에서 확정한다.

## 2. 적용 범위와 결정 상태

### 2.1 적용 범위

- 주요 사용자: 개인사업자와 직원 10인 이하 소규모 법인의 대표자·실무 담당자
- 제품 경로: 자료수집, 기장검토, 부가세, 급여·지급, 연간신고, Path 1 신고 인계
- 중심 진입점: `/dashboard/sebiseo`
- 직접 진입점: 회사 홈과 모든 구조화 작업공간의 사이드바 메뉴
- 책임 경계: 신고 준비와 검토 보조까지 SemuAgent가 담당하고, 최종 신고·제출·납부는 사용자가 수행

### 2.2 결정 상태 표기

| 표기 | 의미 |
|:---|:---|
| **확정** | Product Baseline 또는 승인된 CUI 계약에 이미 반영된 원칙 |
| **운영 기본값** | 이 문서 작성 승인 시 적용하기로 한 제품 기본값 |
| **Open Decision** | 구현 또는 후속 사용자 검증 전에 추가 결정이 필요한 항목 |

이 문서에서 Open Decision으로 표시되지 않은 운영 기본값은 후속 설계의 기준으로 사용한다.
단, 세목별 법정 요건과 계산·제출 규칙은 별도의 기술·QA 문서가 정본이다.

## 3. 핵심 제품 정의

### 3.1 중심 경험

> **세무 메뉴가 있는 앱에 채팅을 붙이는 것이 아니라, 세비서와 일하면 필요한 세무 화면이 그때그때 열린다.**

세비서는 일반 질의응답 챗봇이 아니다. 사용자의 의도를 업무로 만들고, 현재 상태를 확인하며,
다음 행동을 제안하고, 사용자가 판단해야 할 시점에 정확한 구조화 화면을 여는 대화형 업무
지휘 계층이다.

### 3.2 제품 은유

| 요소 | 역할 | 정본 여부 |
|:---|:---|:---|
| 대화 | 조종 장치: 의도 접수, 설명, 질문, 라우팅 | 정본 아님 |
| 상태·진행 카드 | 계기판: 현재 단계, 차단 이유, 다음 행동 | DB 파생 read model |
| 표·원본·snapshot | 검토 표면: 비교, 수정, 명시적 확인 | 사용자 판단 표면 |
| DB·감사 로그 | 실제 상태: canonical 값, 확인자, 시각, 이력 | 정본 |

대화 문장이 DB 상태와 다르면 DB가 우선한다. 세비서는 DB에서 확인되지 않은 성공·완료·Ready를
발화해서는 안 된다.

## 4. 세비서의 네 가지 역할

### 4.1 대화

세비서는 사용자의 자연어 요청을 이해하고 필요한 문맥을 확인한다.

- 세무·회계 업무 요청 접수
- 제품 사용법 설명
- 파일 첨부와 적용 기간 확인 안내
- 처리 결과와 판단 근거 설명
- 사용자가 이해하기 쉬운 다음 행동 제시

대화는 의도를 수집하는 인터페이스일 뿐, 거래·급여·세액·신고자료의 정본이 아니다.

### 4.2 업무 지휘

세비서는 요청을 식별 가능한 업무 문맥에 연결하고, 현재 단계와 차단 요인을 읽어 다음 단계를
선택한다.

- 기존 진행 업무 탐색 또는 신규 업무 시작 제안
- 필요한 파이프라인과 read model 호출
- 현재 단계, 처리 중인 작업, 차단 요인 표시
- 다음 사용자 행동 또는 자동 처리 가능 단계 선택
- 실패·중단·오래된 업무의 복구 경로 제공

업무 지휘는 숨은 canonical mutation을 의미하지 않는다. 신규 업무 생성이나 단계 변경이 실제
데이터를 바꾸는 경우에는 서버 계약, 멱등성, 권한, 감사 기록을 갖춘 명시적 도메인 동작이어야 한다.

### 4.3 검토 연결

세비서는 판단이 필요한 항목을 요약하되, 사용자를 정확한 구조화 검토 표면으로 보낸다.

- 확인 필요 건수와 주요 이유 요약
- 동일 사업자·기간·업무로 필터된 화면 진입
- 원본과 정리값 비교
- AI 제안의 이전 값, 새 값, 근거 표시
- 사용자 수정·확정 후 canonical 상태 재계산

### 4.4 신고 인계

세비서는 모든 준비 조건을 충족한 업무를 Path 1 신고 인계로 연결한다.

- 공식 비암호화 업로드 양식이 확인된 세목: Path 1a 파일 준비
- 공식 양식이 없거나 직접 입력이 필요한 세목: Path 1b `항목 = 값` 준비
- 홈택스·위택스 메뉴, 화면명, 입력 위치 안내
- 사용자의 최종 확인과 직접 제출
- 제출 후 접수증·납부자료 보관과 사후 체크리스트 연결

세비서는 MVP에서 사용자를 대신해 홈택스·위택스에 자동 제출하거나 납부하지 않는다.

## 5. 사용자와 업무 문맥

### 5.1 우선 사용자

운영 기본값은 다음 순서다.

1. 직접 신고를 준비하는 대표자 또는 대표 겸 실무 담당자
2. 직원 10인 이하 회사에서 자료·급여·신고 준비를 담당하는 실무자
3. 기존 세무 소프트웨어 화면에 익숙하고 직접 메뉴 진입을 선호하는 숙련 사용자

세비서는 첫 두 사용자에게 메뉴 학습 비용을 낮춰야 한다. 동시에 세 번째 사용자를 위해
사이드바와 구조화 화면의 직접 진입을 유지한다.

### 5.2 활성 업무 문맥

세비서가 실제 업무를 시작하거나 특정 준비 상태를 단정하려면 최소한 다음 문맥이 필요하다.

| 문맥 | 설명 | 필수 여부 |
|:---|:---|:---:|
| tenant | 권한과 데이터 격리 단위 | 필수 |
| 사업자 | 업무 대상 개인사업자 또는 법인 | 필수 |
| 세목 | 부가세, 원천세, 지급명세서 등 | 신고 업무에 필수 |
| 신고기간 | 월, 반기, 과세기간, 사업연도 | 기간 업무에 필수 |
| 업무유형 | 자료수집, 검토, 신고 준비, 제출 기록 등 | 필수 |
| 사용자 | 요청·검토·확정 행위자 | mutation에 필수 |

문맥이 부족하면 세비서는 완료 상태를 추정하지 않고 확인 질문 또는 구조화 선택 카드를
제시한다.

### 5.3 질문과 구조화 입력의 전환

질문 횟수 자체에 고정 상한을 두지 않는다. 다음 중 하나에 해당하면 자유대화보다 선택 카드,
폼 또는 표로 전환한다.

- 필수 문맥을 자연어만으로 안전하게 확정할 수 없음
- 같은 이름의 사업자·기간·업무 후보가 둘 이상임
- 금액·날짜·세율·공제 여부처럼 정확한 값 입력이 필요함
- 사용자가 여러 후보를 비교해야 함
- 추가 질문보다 구조화 입력이 오류 가능성을 더 낮춤

## 6. 업무 도메인 모델

### 6.1 업무 식별 단위

운영 기본값:

```text
업무 = 사업자 + 세목 + 신고기간 + 업무유형
```

대화방, 브라우저 탭, 업로드 세션, 단일 파일은 업무 자체가 아니다. 이들은 업무와 연결되는
상호작용 또는 처리 단위다.

예시:

```text
사업자: 세무상사
세목: 부가가치세
신고기간: 2026년 제1기 확정
업무유형: 신고 준비
```

### 6.2 논리적 업무 필드

이 표는 제품 개념 계약이며 DB 컬럼을 확정하지 않는다.

| 개념 | 목적 |
|:---|:---|
| 업무 ID | 재진입·로그·알림·멱등성을 위한 안정적 식별자 |
| 업무 키 | 사업자·세목·기간·유형의 중복 판정 기준 |
| 현재 단계 | 업무 흐름에서 사용자가 위치한 단계 |
| 실행 상태 | 대기, 실행 중, 실패 등 처리 상태 |
| 차단 요인 | 다음 단계로 갈 수 없는 기계적 이유 |
| 다음 행동 | 현재 상태에서 가능한 가장 구체적인 행동 |
| 관련 자료 | 업로드 세션, 파일, 원천자료, 계산 snapshot |
| 검토 항목 | 사용자 판단이 필요한 행과 제안 |
| 확인 기록 | 확인자, 시각, 대상 fingerprint, 변경 내용 |
| 최신성 | 원천자료 변경으로 결과가 오래됐는지 여부 |
| 제출 기록 | 직접 제출 확인, 접수증, 납부자료 |

### 6.3 중복 요청과 멱등성

사용자가 같은 업무를 다시 요청하면 기본적으로 기존 활성 업무를 이어간다.

```text
동일 업무 키 + CLOSED 아님
  -> 기존 업무 상태와 다음 행동 표시

동일 업무 키 + CLOSED
  -> 제출 결과 표시 또는 새 수정·재신고 업무 시작 여부 확인

다른 기간·세목·사업자
  -> 별도 업무
```

세비서는 동일한 자연어 요청이 반복됐다는 이유만으로 업로드, 분석, 계산, 확인 mutation을
중복 실행하지 않는다. 실제 구현의 멱등 키와 재시도 정책은 Technical Brief에서 확정한다.

## 7. 업무 상태 모델

### 7.1 주 상태

| 상태 | 의미 | 대표 사용자 화면 |
|:---|:---|:---|
| `DRAFT` | 요청은 있으나 업무 문맥이 확정되지 않음 | 문맥 확인 카드 |
| `CONTEXT_REQUIRED` | 사업자·세목·기간 등 필수 문맥 필요 | 선택 카드 또는 폼 |
| `COLLECTING` | 필요한 자료를 수집 중 | 자료 목록과 누락 안내 |
| `PROCESSING` | 파싱·정규화·계산 등 승인된 파이프라인 실행 중 | 처리 단계와 실제 상태 |
| `REVIEW_REQUIRED` | 사용자 판단 또는 수정이 필요한 항목 존재 | 확인 건수와 검토 CTA |
| `READY` | 세목별 신고 준비 규칙을 모두 통과 | 최종 snapshot과 신고 인계 CTA |
| `HANDED_OFF` | Path 1a 파일 또는 Path 1b 입력값을 사용자에게 인계 | 홈택스·위택스 직접 제출 안내 |
| `SUBMITTED` | 사용자가 직접 제출 사실을 확인하거나 접수증을 등록 | 접수 결과와 납부·사후 행동 |
| `CLOSED` | 제출 후 필수 사후 확인까지 종료 | 읽기 전용 이력 |

`HANDED_OFF`는 신고가 제출됐다는 뜻이 아니다. `SUBMITTED`도 SemuAgent가 자동 제출했다는 뜻이
아니며 사용자의 확인 또는 접수 근거가 존재한다는 뜻이다.

### 7.2 보조 상태

| 상태 | 의미 | 복구 원칙 |
|:---|:---|:---|
| `BLOCKED` | 외부 자료, 사용자 결정, 선행 업무 부족으로 진행 불가 | 차단 이유와 해소 행동 제공 |
| `FAILED` | 실행 단계가 기술적으로 실패 | 성공 데이터 보존, 안전한 재시도 제공 |
| `CANCELLED` | 사용자가 업무 진행을 명시적으로 취소 | 기존 canonical 데이터는 별도 정책에 따라 보존 |
| `STALE` | 원천자료 변경으로 확인·계산 결과가 최신이 아님 | 영향 범위 재검증 후 이전 단계로 이동 |

보조 상태는 주 상태를 대체하기보다 업무의 실행·유효성 조건으로 함께 표현할 수 있다. 실제
저장 모델은 후속 Technical Brief에서 정한다.

경계는 다음과 같이 고정한다. **자료·외부 입력·선행 업무 부족으로 진행할 수 없으면
`BLOCKED`, 현재 행이나 판단 대상에 대한 사용자 검토가 필요하면 `REVIEW_REQUIRED`**다. 하나의
업무에 두 조건이 동시에 존재할 수 있으며, 이때 주 단계와 보조 상태를 함께 표시한다.

### 7.3 기본 상태 전이

```text
DRAFT
  -> CONTEXT_REQUIRED
  -> COLLECTING
  -> PROCESSING
  -> REVIEW_REQUIRED
  -> READY
  -> HANDED_OFF
  -> SUBMITTED
  -> CLOSED
```

모든 업무가 위 상태를 한 번씩 거쳐야 하는 것은 아니다. 이미 문맥이나 자료가 충분하면 앞 단계를
건너뛸 수 있고, 자료가 새로 들어오면 이전 단계로 돌아갈 수 있다. 세목별 허용 전이와 gate는
후속 Readiness Spec과 Technical Brief에서 제한한다.

예외 전이:

```text
PROCESSING -> FAILED -> PROCESSING
COLLECTING 또는 REVIEW_REQUIRED -> BLOCKED -> 이전 주 상태
REVIEW_REQUIRED 또는 READY -> STALE -> PROCESSING 또는 REVIEW_REQUIRED
READY -> REVIEW_REQUIRED        원천자료·판정·확인 상태 변경
HANDED_OFF -> STALE             인계 후 원천자료 변경
모든 미종료 상태 -> CANCELLED   명시적 사용자 취소
```

### 7.4 전이 원칙

- 모든 전이는 서버가 tenant, 사업자, 권한, 현재 상태를 재검증한다.
- 확인 결과를 사용하는 전이는 대상 fingerprint가 일치해야 한다.
- 카드 클릭과 화면 이동은 상태 전이가 아니다.
- 세비서 답변 생성 성공은 도메인 업무 성공이 아니다.
- 실패 후 재시도는 이미 성공한 단계를 중복 적용하지 않는다.
- 원천자료 변경은 영향받는 Ready·인계 결과를 `STALE`로 만들 수 있어야 한다.

## 8. 업무 지휘 루프

세비서는 매 요청과 재진입에서 다음 순서를 따른다.

```text
1. 요청 이해
2. tenant·사업자·세목·기간·업무유형 문맥 확인
3. 동일한 활성 업무와 canonical 상태 조회
4. 현재 단계·차단 요인·최신성 판정
5. 가능한 다음 행동 계산
6. 설명, 확인 질문, 작업 카드 또는 구조화 화면 CTA 표시
7. 명시적 실행이 필요한 경우 사용자 행동 대기
8. 실행 후 DB 상태 재조회
9. 상태 카드와 다음 행동 갱신
```

세비서의 한 응답은 가능한 한 다음 유형 중 하나로 수렴해야 한다.

| 응답 유형 | 사용 시점 |
|:---|:---|
| 설명 | 현재 상태나 계산 근거를 이해시킬 때 |
| 확인 질문 | 업무 필수 문맥이 부족할 때 |
| 작업 카드 | 실행 중·실패·완료한 파이프라인을 보여줄 때 |
| 변경 제안 | 사용자 확인 전 이전 값·새 값·근거를 제시할 때 |
| 화면 이동 | 표·원본·snapshot 검토가 필요할 때 |
| 범위 안내 | 화이트리스트 밖 요청 또는 제품 책임 밖 요청일 때 |

## 9. 진행상황과 차단 요인

### 9.1 진행상황 카드의 다섯 질문

진행상황 카드는 퍼센트보다 다음 질문에 먼저 답해야 한다.

1. 지금 어떤 신고 또는 업무를 준비하고 있는가?
2. 어디까지 처리됐는가?
3. 무엇 때문에 멈춰 있는가?
4. 사용자가 지금 해야 할 한 가지 행동은 무엇인가?
5. 그 행동을 끝내면 Ready가 되는가, 아니면 다음 단계가 남는가?

### 9.2 표시 규칙

- 실제 단계와 건수는 DB read model에서만 파생한다.
- 분모 또는 완료 조건이 불명확하면 임의 퍼센트를 표시하지 않는다.
- 여러 차단 요인이 있으면 심각도와 의존 순서에 따라 대표 행동 하나를 우선한다.
- 사용자가 세부 내용을 열면 전체 차단 목록과 각각의 해결 경로를 볼 수 있어야 한다.
- 공통 법정 일정과 회사의 실제 신고 준비 상태를 혼동해 표시하지 않는다.

### 9.3 차단 요인 모델

각 차단 요인은 최소한 다음 정보를 가져야 한다.

| 개념 | 예시 |
|:---|:---|
| 코드 | `MISSING_REQUIRED_SOURCE` |
| 범위 | 부가세 2026년 제1기 확정 |
| 설명 | 카드 매입자료가 확인되지 않음 |
| 근거 | 필요한 자료 규칙과 현재 수집 상태 |
| 해소 행동 | 카드 내역 업로드 |
| 이동 위치 | 자료수집의 해당 기간·자료유형 |
| 심각도 | 안내, 확인 필요, Ready 차단 |
| 자동 해소 조건 | 유효한 자료가 수집·처리됨 |

사용자에게는 코드보다 이해 가능한 설명을 보여주되, 내부적으로는 안정적인 코드를 사용해
UI·알림·QA가 같은 원인을 가리키게 한다.

## 10. Ready 판정 계약

### 10.1 기본 원칙

`READY`는 LLM의 확률적 판단이나 대화 문장으로 결정하지 않는다. 세목과 기간에 맞는 canonical
검증 규칙을 모두 통과한 결과다.

공통 최소 조건:

- 업무 문맥이 확정됨
- 필수 자료 수집 조건 충족
- 처리 중 또는 실패 상태의 필수 자료 없음
- Ready를 막는 누락·중복·기간 불일치 없음
- 필수 검토 항목 0건
- 계산 또는 신고 snapshot 생성 완료
- 필요한 사용자 확인 완료
- 원천자료와 확인 fingerprint가 최신 상태와 일치
- Path 1 인계에 필요한 세목별 validation 통과

### 10.2 세목별 규칙

공통 조건만으로 실제 세목의 Ready를 확정하지 않는다. 부가세, 원천세, 지급명세서, 연간신고
등은 각 세목의 별도 readiness 규칙과 QA 시나리오를 가져야 한다.

```text
공통 Ready gate
  + 세목별 필수 자료 gate
  + 세목별 계산·검증 gate
  + 사용자 확인 gate
  + 최신 fingerprint gate
  = READY
```

### 10.3 Ready 무효화

다음 사건은 Ready를 자동 재검토 대상으로 만든다.

- 관련 원천 파일 추가·교체·삭제
- 거래 분류 또는 기간 귀속 변경
- 세무 처리 또는 공제 여부 변경
- 급여·직원 기준정보 변경
- 신고 snapshot 생성 후 계산 규칙 또는 입력값 변경
- 확인 대상 fingerprint 불일치

무효화 시 이전 Ready 표시는 유지하지 않고 `STALE` 또는 `REVIEW_REQUIRED`로 되돌린다.

## 11. 대화·카드·표·DB 책임 계약

| 행위 | 대화 | 카드 | 구조화 화면 | DB·서버 |
|:---|:---:|:---:|:---:|:---:|
| 요청 이해 | 주 책임 | 보조 | 아님 | 입력 검증 |
| 현재 상태 표시 | 설명 | 주 책임 | 상세 표시 | 정본·read model |
| 파일 업로드 시작 | 안내·명시적 요청 | 상태 표시 | 자료수집 제공 | 실제 mutation |
| 거래·세액 제안 | 요약 | 건수·CTA | 이전/새 값·근거 | 제안 snapshot |
| 수정 | 금지 | 금지 | 명시적 입력 | 재검증 후 mutation |
| 확정 | 금지 | 금지 | 명시적 확인 | 권한·fingerprint gate |
| Ready 판정 | 설명만 | 표시 | 근거 상세 | 규칙 기반 정본 |
| 신고 인계 | 안내 | CTA | snapshot·파일·값 | validation·이력 |
| 제출 완료 | 단정 금지 | 상태 표시 | 접수 확인·등록 | 제출 근거 저장 |

채팅 안에 버튼이 있더라도 구조화 검토 없이 고위험 mutation을 우회하게 해서는 안 된다.

## 12. 세 가지 핵심 사용자 여정

### 12.1 여정 A: 자료 업로드에서 예외 검토까지

```text
사용자: "6월 카드와 통장 자료 올릴게"
  -> 사업자·적용 기간 확인
  -> 기존 staff_direct 업로드 파이프라인 실행
  -> 실제 파일 처리 상태 카드 표시
  -> 처리 완료 후 확인 필요 항목과 이유 요약
  -> 같은 사업자·기간·세션 또는 관련 행으로 필터된 화면 진입
  -> 원본 비교·수정·확정
  -> canonical 상태 재계산
  -> 다음 자료 또는 신고 준비 행동 제시
```

완료 기준:

- 파일 업로드 성공이 아니라 해당 자료의 처리·검토 상태가 canonical DB에 반영됨
- 확인이 필요한 항목이 있으면 `REVIEW_REQUIRED`
- 파일 처리 실패가 있으면 성공 항목을 보존한 채 복구 행동 제공

### 12.2 여정 B: "지금 무엇이 부족해?"에서 차단 해소까지

```text
사용자: "부가세 준비에 뭐가 부족해?"
  -> 사업자·세목·기간 확정
  -> canonical readiness와 blocker 조회
  -> 부족한 자료·미확인 거래·오류를 우선순위로 표시
  -> 가장 중요한 다음 행동 1개 제시
  -> 사용자가 관련 화면에서 해결
  -> blocker 자동 재계산
  -> 남은 차단 또는 Ready 표시
```

완료 기준:

- 일반적인 체크리스트가 아니라 회사의 실제 상태에 근거함
- blocker가 없을 때만 Ready 검증으로 이동
- 법정 일정만 보고 회사 준비 상태를 추정하지 않음

### 12.3 여정 C: 신고 준비에서 홈택스·위택스 인계까지

```text
사용자: "부가세 신고 준비를 마무리해줘"
  -> 사업자·세목·기간과 활성 업무 확인
  -> Ready gate 실행
  -> 미충족 시 blocker와 검토 화면 제시
  -> 충족 시 최종 snapshot과 사용자 확인
  -> Path 1a 공식 양식 또는 Path 1b 직접입력 값 준비
  -> 사용자가 홈택스·위택스에서 직접 제출
  -> 접수증 또는 제출 사실 등록
  -> 납부·보관·사후 체크리스트 완료 후 CLOSED
```

완료 기준:

- `READY`, `HANDED_OFF`, `SUBMITTED`를 서로 구분
- 양식이 없다는 이유만으로 신고 준비를 막지 않고 Path 1b 제공
- 자동 제출·자격증명 저장·숨은 확인 없음

## 13. 변경 제안과 사용자 확인 계약

### 13.1 변경 제안

AI 또는 규칙 엔진이 canonical 값을 바꾸도록 제안할 때 최소한 다음 정보를 구조화해 표시한다.

- 적용 대상
- 이전 값
- 제안 값
- 제안 이유와 근거
- 예상 영향
- 불확실성 또는 추가 확인 필요 여부

여러 행을 일괄 적용할 때는 적용 범위, 제외 항목, 총 영향과 되돌리기 가능 여부를 먼저 보여준다.

### 13.2 사용자 확인

확인은 다음 조건을 모두 만족해야 유효하다.

- 사용자가 확인하는 대상과 범위를 볼 수 있음
- 확인 시점의 fingerprint가 저장됨
- 확인자와 확인 시각이 기록됨
- 서버가 tenant, 권한, 현재 상태를 재검증함
- 확인 직전 대상이 바뀌었으면 적용하지 않고 재검토를 요구함

단순히 대화에서 "네", "진행해줘"라고 말하는 것은 거래·급여·세액·신고 snapshot의 최종
확정으로 사용하지 않는다. 저위험 작업 실행 여부와 고위험 canonical 확인은 별도 계약으로 구분한다.

### 13.3 취소와 되돌리기

- 화면 이동이나 대화 취소는 이미 완료된 canonical mutation을 자동 취소하지 않는다.
- 되돌릴 수 있는 변경은 감사 이력을 보존한 새 변경으로 처리한다.
- 제출·접수처럼 외부 시스템에서 완료된 행위는 앱 내부 undo로 되돌렸다고 표현하지 않는다.
- CUI-5에서 mutation 유형별 undo 가능 범위를 별도 확정한다.

## 14. 세비서 발화 계약

### 14.1 허용되는 완료 표현

| canonical 상태 | 허용 표현 예시 |
|:---|:---|
| 처리 요청만 접수 | "작업을 시작했습니다." |
| 일부 처리 중 | "자료를 처리하고 있습니다." |
| 검토 필요 | "7건을 확인해야 다음 단계로 갈 수 있습니다." |
| Ready gate 통과 | "2026년 제1기 부가세 신고자료가 준비 가능한 상태입니다." |
| Path 1 인계 완료 | "홈택스에서 확인·제출할 값이 준비되었습니다." |
| 제출 근거 등록 | "제출 접수 정보를 보관했습니다." |

`READY`에서도 "신고가 완료되었습니다"라고 말하지 않는다. 해당 표현은 제출 근거가 있는
`SUBMITTED` 이후에만 제한적으로 사용할 수 있으며, 납부 완료와도 구분한다.

### 14.2 금지 표현과 행동

- DB 검증 전 "완료되었습니다", "문제없습니다", "신고할 수 있습니다" 단정
- 실행되지 않은 작업을 "처리했습니다"라고 표현
- 추정·일반 안내를 회사의 실제 상태처럼 표현
- 세비서를 세무사·세무대리인·세무회계 전문가로 표현
- 불확실한 세무 판단을 확정값처럼 표현
- 실패를 숨기고 성공 항목만으로 전체 성공처럼 요약
- 자동 신고·자동 납부가 이뤄진 것처럼 표현
- 비밀번호·인증서·계좌·카드 비밀정보를 대화 기록에 남기도록 유도

### 14.3 응답 구성 원칙

- 먼저 현재 결과 또는 상태를 말한다.
- 상태의 근거와 범위를 명시한다.
- 불확실성 또는 차단 요인이 있으면 숨기지 않는다.
- 가능한 경우 가장 구체적인 다음 행동 하나를 제시한다.
- 상세 검토가 필요하면 필터된 구조화 화면으로 연결한다.
- 화이트리스트 밖 요청은 짧게 거절하고 가능한 제품 범위를 안내한다.

## 15. 실패·중단·재시도·재개

### 15.1 실패 원칙

- 성공한 기존 데이터와 확정 상태를 보존한다.
- 실패한 단계, 영향 범위, 재시도 가능 여부를 구분한다.
- 자동 재시도와 사용자 재시도를 구분해 중복 실행을 방지한다.
- 기술 오류를 세무 검토 오류처럼 표현하지 않는다.
- 복구할 수 없으면 직접 메뉴와 수동 처리 경로를 제공한다.

### 15.2 재진입

사용자가 앱을 다시 열면 빈 대화부터 강제하지 않는다. canonical 업무 상태를 기준으로 다음을
보여준다.

- 가장 가까운 신고 일정
- 진행 중이거나 차단된 우선 업무
- 마지막으로 갱신된 상태
- 지금 필요한 다음 행동
- 최근 업로드 결과

대화 이력 보존 여부와 무관하게 업무는 재개할 수 있어야 한다. CUI-3의 ephemeral 대화 이력은
업무 복구의 정본으로 사용하지 않는다.

### 15.3 오래된 업무

장기간 갱신되지 않은 업무는 자동으로 완료 처리하지 않는다. 법정 일정, 기간 종료, 원천자료
변경 여부를 다시 확인한 후 다음 중 하나를 제시한다.

- 기존 업무 계속
- 새 기간 업무 시작
- 수정·재신고 업무로 분기
- 명시적 취소 또는 보관

## 16. 직접 메뉴 진입과 장애 복구

세비서는 중심 진입점이지만 유일한 진입점이 아니다.

- 사이드바에서 회사 홈과 모든 작업공간으로 직접 이동할 수 있어야 한다.
- 숙련 사용자는 세비서 질문 없이 기간·업무 화면을 열 수 있어야 한다.
- AI provider 장애가 있어도 자료수집·검토·신고 준비 화면은 사용할 수 있어야 한다.
- 세비서가 요청을 이해하지 못하면 관련 메뉴 후보를 제공한다.
- 세비서 카드와 상세 화면 상태가 다르면 상세 화면과 DB를 우선하고 카드를 다시 계산한다.

이 원칙은 숙련 사용자의 효율, AI 장애 시 복구, 잘못된 의도 분류의 탈출 경로를 동시에 보장한다.

## 17. 알림과 업무 재호출

알림은 대화를 다시 열게 하는 목적이 아니라 해결해야 할 실제 업무로 복귀시키는 수단이다.

알림에 필요한 최소 정보:

- 사업자
- 세목과 기간
- 현재 단계 또는 차단 이유
- 필요한 행동
- 마감과의 관계
- 정확한 업무 또는 구조화 화면 링크

알림을 클릭했을 때 새 업무를 중복 생성하지 않고 기존 업무로 진입해야 한다.

### Open Decision OD-01: 백그라운드 완료 통지 채널

- 앱 내 알림만 우선할지
- 이메일 또는 외부 메시지 채널을 MVP에 포함할지
- 처리 완료, 차단 발생, 마감 임박 중 어떤 사건을 통지할지

후속 알림 Brief에서 사용자 피로도와 민감정보 노출 기준을 함께 확정한다.

## 18. 개인정보·보안·감사

- 모든 회사 범위 조회와 mutation은 tenant와 사업자를 격리한다.
- 원천 금융·급여·세무자료를 불필요하게 LLM provider에 전송하지 않는다.
- 파일 바이트, Blob URL, storage key, 인증정보는 대화 history에 포함하지 않는다.
- 주민등록번호, 계좌·카드번호, 비밀번호, 토큰 등은 provider 호출 전후 redaction한다.
- 홈택스·위택스 비밀번호, 공동인증서, 금융기관 자격증명 원문을 저장하지 않는다.
- 사용자 확인, 변경, 인계, 제출 근거는 대화 이력과 별도의 감사 로그에 남긴다.
- 감사 로그는 대상, 행위자, 시각, 이전·새 값, fingerprint, 결과를 추적할 수 있어야 한다.

대화 편의성은 데이터 최소 전송, tenant 격리, 명시적 확인보다 우선하지 않는다.

## 19. MVP 포함·제외 범위

### 19.1 포함

- 세비서 첫 화면과 기존 사이드바 병행
- 화이트리스트 대화와 제품 사용 안내
- 파일 첨부 전 사업자·기간 확인
- 기존 자료수집·분석·도메인 파이프라인 재사용
- 실제 DB 상태에서 파생한 진행·결과·차단 카드
- 필터된 구조화 검토 화면 연결
- 세목별 Ready 규칙과 최신성 검증
- Path 1a 또는 1b 신고 인계
- 사용자 직접 제출 확인과 접수증 보관
- 변경·확인·감사 기록

세목별 Ready 계약은 **부가세를 첫 적용 대상**으로 검증한다. 부가세에서 공통 gate와 세목별
gate의 분리를 확인한 뒤 원천세·지급명세서·연간신고로 확장한다.

### 19.2 제외

- 범용 ChatGPT형 잡담·코딩·일반 지식 응답
- 제품 범위 밖의 포괄적 세무상담 또는 세무대리
- 채팅만으로 거래·급여·세액·신고자료 확정
- 숨은 mutation과 자동 일괄 승인
- 자동 홈택스·위택스 신고와 자동 납부
- 홈택스·은행·카드·인증서 자격증명 원문 저장
- 기존 메뉴와 구조화 작업공간 제거
- 대화 로그를 회계 원장 또는 감사 정본으로 사용
- 음성, 장기 자율 에이전트, 임의 플러그인 실행을 MVP 필수요건으로 포함

## 20. 성공 지표

세비서의 성공을 대화량이나 답변 길이로 평가하지 않는다.

| 지표 | 목적 |
|:---|:---|
| 첫 유효 결과까지 걸린 시간 | 사용자가 초기 자료·상태·다음 행동을 얻는 속도 |
| 업무 문맥 확정 성공률 | 사업자·세목·기간 오류 없이 업무가 시작되는 비율 |
| 시작 전 확인 질문 수 | 대화 진입 비용과 문맥 설계 품질 |
| 다음 행동 실행률 | 카드와 안내가 실제 업무 해결로 이어지는 정도 |
| 검토 항목 해결률 | 구조화 화면으로 연결된 예외가 확정되는 비율 |
| 차단 상태 이탈률 | blocker를 본 뒤 포기하는 비율 |
| Ready 판정 번복률 | 최신성·규칙·확인 계약의 안정성 |
| 잘못 실행·중복 실행 비율 | 지휘·멱등성 안전성 |
| 신고 인계 완료율 | Ready에서 Path 1a/1b 결과까지 도달하는 비율 |
| 제출 근거 보관률 | 직접 제출 후 접수 이력이 남는 비율 |
| 직접 메뉴 복구 성공률 | AI 장애·오분류 상황에서 작업을 계속할 수 있는 정도 |

초기 MVP에서는 첫 유효 결과 시간, 다음 행동 실행률, 검토 항목 해결률, 신고 인계 완료율,
잘못 실행 비율을 우선 측정한다.

## 22. Open Decisions

### OD-01. 백그라운드 완료 통지 채널

앱 내 알림, 이메일, 외부 메시지 채널의 MVP 우선순위와 민감정보 표시 범위를 결정해야 한다.

### OD-02. 저위험 대화 실행의 허용 범위

자료 재조회, read-only 검사, 기존 분석 재시도처럼 대화에서 바로 시작할 수 있는 저위험 작업과
항상 구조화 확인이 필요한 mutation의 허용목록을 CUI-5 전에 확정해야 한다.

**CUI-5 운영 기본값:** 세비서는 read-only 상태 조회와 화면 이동만 수행한다. 재분석·재시도·수정·
확정은 해당 구조화 화면에서 수행하며, 채팅·카드에서 canonical mutation을 시작하지 않는다.
향후 저위험 실행 허용목록 확대는 별도 승인한다.

### OD-03. 복수 활성 업무의 우선순위

동시에 여러 세목·기간 업무가 열려 있을 때 마감일, blocker 심각도, 사용자 최근 행동 중 무엇을
우선순위의 주 기준으로 삼을지 정해야 한다.

### OD-04. 대화 보존 정책의 후속 범위

CUI-3의 탭 단위 ephemeral history를 유지할지, canonical 업무와 분리된 장기 대화 이력을 선택적으로
도입할지 개인정보·고객지원·사용성 검증 후 결정한다.

### OD-05. 수정·재신고 업무 모델

이미 `SUBMITTED` 또는 `CLOSED`인 기간에 자료가 바뀌었을 때 기존 업무를 재개할지, 별도 수정·재신고
업무를 생성할지 세목별 법적·운영 요구와 함께 정의한다.

## 23. 후속 설계 산출물

이 문서 승인 후 다음 순서로 파생 문서를 만든다.

1. **CUI-4 마감**: 브라우저·업로드 직후 갱신·자료수집 회귀와 문서 동기화 완료
2. **UI_Screens**: 업로드 → 거래 검토 → 확정 → 상태 갱신의 첫 세로 여정 Preview
3. **CUI-4d Brief**: 거래 단위 확인 필요 read model과 필터된 자료대조원장 진입 계약
4. **CUI-5 Brief**: 변경 제안, fingerprint, 명시적 확인, undo, 감사와 상태 피드백 계약
5. **단계별 구현**: CUI-4d read-only 연결 후 CUI-5 구조화 확정 세로 흐름 구현
6. **세목별 Readiness Spec**: 부가세부터 공통·세목별 Ready 규칙 정의
7. **QA와 CUI-6**: tenant 격리, 중복 요청, stale, 실패 복구, 금지 발화, E2E·성능·접근성
8. **Backlog Context Lock**: 각 구현 단위를 위 문서와 Preview에 연결

상세 UI 또는 DB 구조를 이 문서만 보고 바로 구현하지 않는다. HTML UI Preview Gate와 Pre-Code
Technical Brief를 통과한 뒤 구현한다.

## 24. 핵심 용어

| 용어 | 정의 |
|:---|:---|
| 세비서 | 세무·회계 업무를 도와주는 대화형 업무 지휘 계층 |
| 업무 | 사업자·세목·신고기간·업무유형으로 식별되는 진행 단위 |
| canonical | 제품이 실제 상태로 신뢰하는 DB 및 감사 기록 |
| read model | canonical 데이터를 화면 목적에 맞게 읽기 전용으로 가공한 상태 |
| blocker | 다음 단계 또는 Ready를 막는 구체적 조건 |
| Ready | 세목별 기계적 준비 규칙을 모두 통과한 상태 |
| fingerprint | 사용자가 확인한 대상이 이후 변경되지 않았음을 검증하는 식별값 |
| snapshot | 특정 시점의 계산·신고 준비 결과를 고정해 검토하는 단위 |
| Path 1a | 공식 비암호화 양식 파일을 준비해 사용자가 직접 업로드하는 경로 |
| Path 1b | 신고 화면에 직접 입력할 `항목 = 값`을 제공하는 경로 |
| handoff | 준비 결과를 사용자 또는 향후 연결된 외부 주체에게 명시적으로 인계하는 단계 |
